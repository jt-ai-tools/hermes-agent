# 動畫設計思維

在編寫任何程式碼之前，如何決定要動畫化什麼內容以及如何組織結構。

## 我應該將這個動畫化嗎？

並非所有事物都能從動畫中受益。動態會增加認知負荷。糟糕的動畫比良好的靜態圖表更糟。

**在以下情況進行動畫處理：**
- 一個序列隨時間展開（演算法步驟、推導過程、管線階段）
- 空間關係發生變化（轉換、變形、旋轉）
- 某物是由多個部分組成的（建構、組裝、累積）
- 正在比較不同狀態（之前/之後，方法 A 與方法 B）
- 時間演化是重點（訓練曲線、波傳導、梯度下降）

**在以下情況顯示靜態內容：**
- 該概念是一個單一的標記圖表（電路、解剖結構、架構概覽）
- 動態會干擾空間佈局
- 觀看者需要仔細研究它（密集的表格、參考圖表）
- 透過標記良好的圖表，該概念已經很直觀

**經驗法則：** 如果你會用「首先是 X，然後是 Y，最後是 Z」來解釋它——請使用動畫。如果你會透過指向一張圖片的各個部分來解釋它——請使用靜態顯示。

## 將概念分解為動畫

### 步驟 1：先寫下旁白

在撰寫任何程式碼之前，先寫下敘述者會說的話。這決定了：
- **順序** — 哪個概念先出現
- **時長** — 每個想法分配到多少時間
- **視覺效果** — 當觀看者聽到每個句子時必須「看見」什麼

當旁白說「梯度指向山上」時，場景必須在那個時刻顯示一個梯度箭頭。如果視覺效果與音訊不符，觀看者的腦部會分散注意力，導致兩條線索都無法被有效接收。

### 步驟 2：識別視覺節奏（Visual Beats）

一個「節奏（Beat）」是螢幕上發生變化的時刻。在你的旁白中標記每個節奏：

```
「考慮一個函數 f(x)。」          → [節奏：坐標軸 + 曲線出現]
「在這一點上...」                → [節奏：點出現在曲線上]
「...斜率是正的。」              → [節奏：繪製切線]
「所以梯度告訴我們向左移動。」    → [節奏：箭頭指向左側，點移動]
```

每個節奏對應一個 `self.play()` 呼叫或一組同時進行的動畫。

### 步驟 3：為每個節奏選擇正確的工具

| 視覺需求 | Manim 做法 |
|-------------|----------------|
| 物件首次出現 | `Create`, `Write`, `FadeIn`, `GrowFromCenter` |
| 物件轉換為另一個 | `Transform`, `ReplacementTransform`, `FadeTransform` |
| 注意力吸引到現有物件 | `Indicate`, `Circumscribe`, `Flash`, `ShowPassingFlash` |
| 保持連續關係 | `add_updater`, `always_redraw`, `ValueTracker` |
| 物件離開場景 | `FadeOut`, `Uncreate`, `ShrinkToCenter` |
| 保持可見的靜態背景 | `self.add()`（無動畫） |

## 節奏：普遍的錯誤是太快

### 時間規則

| 內容類型 | 螢幕上最少停留時間 |
|-------------|----------------------|
| 新方程式出現 | 2.0s 動畫 + 2.0s 停頓 |
| 新概念標籤 | 1.0s 動畫 + 1.0s 停頓 |
| 關鍵見解（「恍然大悟」時刻） | 2.5s 動畫 + 3.0s 停頓 |
| 輔助註釋 | 0.8s 動畫 + 0.5s 停頓 |
| 場景轉換（淡出所有內容） | 0.5s 動畫 + 0.3s 停頓 |

### 留白與緩衝（Breathing Room）

在每次揭露之後，加入 `self.wait()`。觀看者需要時間來：
1. 閱讀新文字
2. 將其與螢幕上已有的內容聯繫起來
3. 對接下來會發生的事情形成預期

**沒有等待 = 觀看者永遠落後於你。** 當你已經開始變形方程式時，他們可能還在閱讀它。

### 節奏變化

單調的節奏感覺就像在聽課。請改變節奏：
- **緩慢構建** 核心概念（長 `run_time`，長停頓）
- **快速連續** 輔助細節（短 `run_time`，最小停頓）
- **戲劇性停頓** 在關鍵揭露之前（在「恍然大悟」前增加額外的 `self.wait(2.0)`）
- **快速蒙太奇** 用於「這適用於 X, Y, Z...」序列（使用緊密的 `lag_ratio` 配合 `LaggedStart`）

## 旁白同步

### 「先見後聞」原則

視覺效果應該在旁白描述它之前「稍微」出現。當觀看者看到一個圓圈出現，然後聽到「考慮一個圓圈」時，視覺效果會預先啟發他們的大腦接受這個概念。反之——先聽後見——會產生混淆，因為他們正在螢幕上尋找尚未出現的東西。

### 實際的時間安排

```python
# 場景時長應與旁白時長相匹配。
# 如果此場景的旁白為 8 秒：
# 總動畫 run_times + 總 self.wait() 時間 = ~8 秒。

# 使用 manim-voiceover 進行自動同步：
with self.voiceover(text="梯度指向山下") as tracker:
    self.play(GrowArrow(gradient_arrow), run_time=tracker.duration)
```

## 方程式分解策略

### 「變暗與揭露」模式

逐步構建複雜方程式時：
1. 顯示完整方程式，並將透明度設定為 `opacity=0.2`（設定對目標的預期）
2. 以全透明度強調第一個項（Term）
3. 解釋它
4. 強調下一個項，將第一個項變暗至 `0.5`（現在它是背景脈絡）
5. 重複直到完整方程式變亮

這比由左至右構建更好，因為觀看者始終能看到終點。

### 項目的順序

按照觀看者需要「理解」它們的順序來動畫化各個項，而不是按照它們在方程式中出現的順序。對於 `E = mc²`：
- 顯示 `E`（我們想知道的東西）
- 然後是 `m`（輸入）
- 然後是 `c²`（使其運作的常數）
- 最後是 `=`（將它們連接起來）

## 架構與管線圖

### 方框細微度

最常見的錯誤：方框太多。每個方框都是觀看者必須追踪的一個概念。五個帶有清晰標籤的方框勝過十二個帶有縮寫的方框。

**規則：** 如果兩個連續的方框可以標記為「X」和「處理 X 的輸出」，請將它們合併為一個方框。

### 動畫策略

由左至右（或由上至下）構建管線，並用箭頭連接：
1. 第一個方框單獨出現 → 解釋它
2. 箭頭從第一個方框長到第二個 → 「輸出餵入到...」
3. 第二個方框出現 → 解釋它
4. 重複

然後顯示數據流過：沿著箭頭使用 `ShowPassingFlash`，或使用彩色小點遍歷路徑。

### 「縮放與返回」模式

對於複雜系統：
1. 顯示完整概覽（所有方框，縮小顯示）
2. 縮放到一個方框（使用 `MovingCameraScene.camera.frame.animate`）
3. 將該方框展開為其內部組件
4. 縮回概覽
5. 縮放到下一個方框

## 常見的設計錯誤

1. **同時動畫化所有內容。** 觀看者一次只能追踪 1-2 個同時進行的動畫。超過這個數量，大腦將無法記錄任何資訊。
2. **缺乏視覺層次。** 所有內容都使用相同的透明度/大小/顏色，這意味著沒有任何重點突出。請使用透明度分層。
3. **沒有背景脈絡的方程式。** 單獨出現一個方程式沒有意義。務必先顯示幾何/視覺解釋，或者同步顯示。
4. **忽略「為什麼」。** 顯示了轉換「如何」運作，但沒有說明「為什麼」它很重要。增加一個句子/標籤來解釋目的。
5. **全程節奏完全一致。** 每個動畫 `run_time` 都是 1.5，每個等待都是 1.0。請靈活變化。
6. **忘記受眾。** 面向高中生的影片與面向博士生的影片，在節奏和複雜度上需求不同。在規劃階段就決定受眾。
