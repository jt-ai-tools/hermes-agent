# ASCII 影片生產流程 (ASCII Video Production Pipeline)

## 創意標準

這是一門視覺藝術。ASCII 字元是媒介；電影感是標準。

**在編寫任何程式碼之前**，請先明確創意概念。氛圍是什麼？這在講述什麼視覺故事？是什麼讓「這個」專案與其他所有 ASCII 影片不同？使用者的提示詞只是一個起點 —— 請以創意的野心來詮釋它，而非刻板的轉述。

**首刷即卓越是不可妥協的。** 輸出結果必須在視覺上極具衝擊力，且無需反覆修改。如果看起來平庸、單調，或像「AI 生成的 ASCII 藝術」，那就是錯誤的 —— 請在交付前重新思考創意概念。

**超越參考詞彙。** 參考資料中的效果清單、著色器預設和調色盤程式庫只是基礎詞彙。對於每個專案，請嘗試組合、修改並發明新的模式。目錄就像是調色盤 —— 而您負責繪製整幅畫。

**保持積極主動的創意。** 當專案需要時，請擴展技能的詞彙。如果參考資料中沒有願景所需的內容，請自行構建。至少包含一個使用者沒有要求但會欣賞的視覺時刻 —— 一個過場、一個效果或一個提升整體質感的顏色選擇。

**凝聚的美學優於技術的正確性。** 影片中的所有場景必須透過統一的視覺語言連接起來 —— 共用的色溫、相關的字元調色盤、一致的動作語彙。一個技術正確但每個場景都使用隨機不同效果的影片，在美學上是失敗的。

**密集、層次分明、深思熟慮。** 每一格畫面都應該讓觀看者感到值得。絕不要使用純黑的背景。務必使用多網格構圖。務必包含場景間的變化。務必使用有意的色彩。

## 技術棧 (Stack)

每個專案使用單一且自包含的 Python 腳本。不需要 GPU。

| 層級 | 工具 | 用途 |
|-------|------|---------|
| 核心 | Python 3.10+, NumPy | 數學運算、陣列操作、向量化效果 |
| 信號 | SciPy | FFT、峰值檢測（音訊模式） |
| 影像 | Pillow (PIL) | 字體點陣化、影格解碼、影像 I/O |
| 影片 I/O | ffmpeg (CLI) | 解碼輸入、編碼輸出、音軌混流 (Mux) |
| 並行 | concurrent.futures | 使用 N 個工作進程進行批次/片段渲染 |
| TTS | ElevenLabs API (選用) | 生成旁白片段 |
| 選用 | OpenCV | 影片影格取樣、邊緣檢測 |

## 流程架構

每種模式都遵循相同的 6 階段流程：

```
INPUT → ANALYZE → SCENE_FN → TONEMAP → SHADE → ENCODE
```

1. **INPUT (輸入)** —— 載入/解碼原始素材（影片影格、音訊樣本、影像或無）
2. **ANALYZE (分析)** —— 提取每格特徵（音頻頻段、影片亮度/邊緣、運動向量）
3. **SCENE_FN (場景函式)** —— 場景函式渲染到像素畫布 (`uint8 H,W,3`)。透過 `_render_vf()` + 像素混合模式組合多個字元網格。請參閱 `references/composition_zh_TW.md`
4. **TONEMAP (色調映射)** —— 基於百分位數的自適應亮度歸一化。請參閱 `references/composition_zh_TW.md` § 自適應色調映射
5. **SHADE (著色)** —— 透過 `ShaderChain` + `FeedbackBuffer` 進行後處理。請參閱 `references/shaders_zh_TW.md`
6. **ENCODE (編碼)** —— 將原始 RGB 影格透過管道傳送至 ffmpeg 進行 H.264/GIF 編碼

絕不要在整段影片中使用相同的配置。針對每個章節/場景：
- **使用不同的背景效果**（或組合 2-3 種）

## 工作流程

### 步驟 1：創意願景

在編寫任何程式碼之前，請先明確創意概念：

- **情緒/氛圍**：觀眾應該感受到什麼？充滿活力的、沉思的、混亂的、優雅的、不祥的？
- **視覺故事**：在整個時長內發生了什麼？建立張力？轉化？溶解？
- **色彩世界**：暖色調還是冷色調？單色？霓虹？大地色？主導色調是什麼？
- **字元紋理**：密集的數據？稀疏的星星？有機的點？幾何塊？
- **是什麼讓「這個」與眾不同**：是什麼讓這個專案獨一無二？
- **情感弧線**：場景如何推進？以能量開場，建立高潮，最後解決？

將使用者的提示詞對應到美學選擇。一個「冷靜的 lo-fi 視覺化工具」所需的一切都與「故障風格的網路龐克數據流」截然不同。

### 步驟 2：技術設計

- **模式** —— 以上 6 種模式中的哪一種
- **解析度** —— 橫向 1920x1080（預設）、縱向 1080x1920、正方形 1080x1080 @ 24fps
- **硬體檢測** —— 自動檢測核心/記憶體，設定品質配置文件。請參閱 `references/optimization_zh_TW.md`
- **章節** —— 將時間戳記映射到場景函式，每個函式都有自己的效果/調色盤/顏色/著色器配置
- **輸出格式** —— MP4（預設）、GIF (640x360 @ 15fps)、PNG 序列

### 步驟 3：構建腳本

單一 Python 檔案。組件（附參考資料）：

1. **硬體檢測 + 品質配置文件** —— `references/optimization_zh_TW.md`
2. **輸入載入器** —— 取決於模式；`references/inputs_zh_TW.md`
3. **特徵分析器** —— 音訊 FFT、影片亮度或合成
4. **網格 + 渲染器** —— 帶有位圖快取的多密度網格；`references/architecture_zh_TW.md`
5. **字元調色盤** —— 每個專案多個；`references/architecture_zh_TW.md` § 調色盤
6. **色彩系統** —— HSV + 離散 RGB + 和諧色生成；`references/architecture_zh_TW.md` § 色彩
7. **場景函式** —— 每個函式返回 `canvas (uint8 H,W,3)`；`references/scenes_zh_TW.md`
8. **色調映射 (Tonemap)** —— 自適應亮度歸一化；`references/composition_zh_TW.md`
9. **著色器流水線** —— `ShaderChain` + `FeedbackBuffer`；`references/shaders_zh_TW.md`
10. **場景表 + 調度器** —— 時間 → 場景函式 + 配置；`references/scenes_zh_TW.md`
11. **並行編碼器** —— 使用 ffmpeg 管道進行 N 個工作進程的片段渲染
12. **Main (主程式)** —— 協調完整流程

### 步驟 4：品質驗證

- **先測試影格**：在完整渲染前，先渲染關鍵時間戳記的單一影格
- **亮度檢查**：所有 ASCII 內容的 `canvas.mean() > 8`。如果太暗，請降低 Gamma
- **視覺連貫性**：所有場景是否感覺屬於同一個影片？
- **創意願景檢查**：輸出結果是否符合步驟 1 的概念？如果看起來很平庸，請重做

## 關鍵實作注意事項

### 亮度 —— 使用 `tonemap()`，而非線性倍數

這是排名第一的視覺問題。黑底上的 ASCII 本質上是暗的。**絕不要使用 `canvas * N` 倍數** —— 它們會剪裁高光。請使用自適應色調映射：

```python
def tonemap(canvas, gamma=0.75):
    f = canvas.astype(np.float32)
    lo, hi = np.percentile(f[::4, ::4], [1, 99.5])
    if hi - lo < 10: hi = lo + 10
    f = np.clip((f - lo) / (hi - lo), 0, 1) ** gamma
    return (f * 255).astype(np.uint8)
```

流程：`scene_fn() → tonemap() → FeedbackBuffer → ShaderChain → ffmpeg`

每場景 Gamma 值：預設 0.75，曝光過度 (Solarize) 0.55，海報化 (Posterize) 0.50，明亮場景 0.85。對於深色圖層，請使用 `screen` 混合（而非 `overlay`）。

### 字體單元高度 (Font Cell Height)

macOS Pillow：`textbbox()` 返回的高度錯誤。請使用 `font.getmetrics()`：`cell_height = ascent + descent`。請參閱 `references/troubleshooting_zh_TW.md`。

### ffmpeg 管道死鎖

對於長時間運行的 ffmpeg，絕不要使用 `stderr=subprocess.PIPE` —— 緩衝區會在 64KB 時填滿並導致死鎖。請重定向到檔案。請參閱 `references/troubleshooting_zh_TW.md`。

### 字體相容性

並非所有 Unicode 字元都能在所有字體中渲染。請在初始化時驗證調色盤 —— 渲染每個字元，檢查是否有空白輸出。請參閱 `references/troubleshooting_zh_TW.md`。

### 每片段架構 (Per-Clip Architecture)

對於分段影片（引用、場景、章節），請將每個段落渲染為獨立的片段檔案，以便進行並行渲染和選擇性重新渲染。請參閱 `references/scenes_zh_TW.md`。

## 創意發散（僅在使用者要求實驗性/創意/獨特輸出時使用）

如果使用者要求實驗性、令人驚訝或非傳統的輸出，請選擇最合適的策略，並在生成程式碼之前先理清步驟。

- **強制關聯 (Forced Connections)** —— 當使用者想要跨領域靈感時（「讓它看起來很有機」、「工業美學」）
- **概念融合 (Conceptual Blending)** —— 當使用者指名要結合兩件事物時（「海洋遇上音樂」、「太空 + 書法」）
- **側向思維 (Oblique Strategies)** —— 當使用者極度開放時（「給我驚喜」、「我從未見過的東西」）

### 強制關聯
1. 挑選一個與視覺目標無關的領域（天氣系統、微生物學、建築、流體力學、紡織編織）
2. 列出其核心視覺/結構元素（侵蝕 → 逐漸顯現；有絲分裂 → 分裂複製；編織 → 交織圖案）
3. 將這些元素對應到 ASCII 字元和動畫模式
4. 綜合 —— 在字元網格中，「侵蝕」或「結晶」看起來像什麼？

### 概念融合
1. 命名兩個不同的視覺/概念空間（例如：海浪 + 樂譜）
2. 建立對應關係（波峰 = 高音，波谷 = 休止符，泡沫 = 斷奏）
3. 有選擇地融合 —— 保留最有趣的映射，捨棄生硬的映射
4. 開發僅存在於融合中的突發屬性

### 側向思維
1. 抽一張卡片：「尊重你的錯誤，將其視為隱藏的意圖」/「使用舊點子」/「你最好的朋友會怎麼做？」/「強調缺陷」/「把它倒過來」/「只取一部分，而非整體」/「反轉」
2. 針對目前的 ASCII 動畫挑戰詮釋該指令
3. 在編寫程式碼之前，將側向洞察應用於視覺設計
