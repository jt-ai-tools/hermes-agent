# 系統化偵錯 (Systematic Debugging)

## 概述

隨機的修復只是在浪費時間，且會產生新的 Bug。倉促的補丁只會掩蓋底層問題。

**核心原則：** 在嘗試修復之前，「務必」找出根因 (root cause)。僅針對症狀進行修復是失敗的做法。

**違反此流程的步驟，就是違反偵錯的精神。**

## 鋼鐵法則 (The Iron Law)

```
先進行根因調查，否則不准修復
```

如果你尚未完成階段 1，就不能提出修復方案。

## 何時使用

適用於「任何」技術問題：
- 測試失敗
- 正式環境的 Bug
- 非預期行為
- 效能問題
- 建置 (build) 失敗
- 整合問題

**尤其在以下情況務必使用：**
- 面臨時間壓力時（緊急情況常讓人想用猜的）。
- 當「只要一個快速修復」看起來很明顯時。
- 你已經嘗試過多種修復方法。
- 先前的修復無效。
- 你尚未完全理解問題。

**不可跳過的情況：**
- 問題看起來很簡單（簡單的 Bug 也有根因）。
- 你很趕時間（匆忙會保證你需要重做）。
- 有人要求「現在」就修好（系統化方法比無頭蒼蠅式的亂試更快）。

## 四個階段

你「必須」在完成前一個階段後，才能進入下一個階段。

---

## 階段 1：根因調查 (Root Cause Investigation)

**在嘗試「任何」修復之前：**

### 1. 仔細閱讀錯誤訊息

- 不要跳過錯誤或警告。
- 它們通常包含了確切的解決方案。
- 完整閱讀堆疊追蹤 (stack traces)。
- 記錄行號、檔案路徑、錯誤代碼。

**行動：** 對相關原始碼檔案使用 `read_file`。使用 `search_files` 在程式碼庫中尋找錯誤字串。

### 2. 穩定重現問題

- 你能可靠地觸發它嗎？
- 確切的步驟是什麼？
- 每次都會發生嗎？
- 如果無法重現 → 收集更多資料，不要瞎猜。

**行動：** 使用 `terminal` 工具執行失敗的測試或觸發 Bug：

```bash
# 執行特定的失敗測試
pytest tests/test_module.py::test_name -v

# 執行並顯示詳細輸出
pytest tests/test_module.py -v --tb=long
```

### 3. 檢查最近的變更

- 發生了什麼變更可能導致此問題？
- 查看 Git diff、最近的 commit。
- 新的依賴項目、配置變更。

**行動：**

```bash
# 最近的 commit
git log --oneline -10

# 尚未 commit 的變更
git diff

# 特定檔案的變更記錄
git log -p --follow src/problematic_file.py | head -100
```

### 4. 在多組件系統中收集證據

**當系統包含多個組件時（API → 服務 → 資料庫，或 CI → 建置 → 部署）：**

**在提出修復方案之前，先加入診斷工具 (diagnostic instrumentation)：**

針對「每個」組件邊界：
- 記錄進入該組件的資料。
- 記錄離開該組件的資料。
- 驗證環境變數/配置的傳遞。
- 檢查每一層的狀態。

執行一次以收集證據，找出問題「在哪裡」發生中斷。
接著分析證據以識別故障組件。
最後針對該特定組件進行調查。

### 5. 追蹤資料流 (Data Flow)

**當錯誤發生在呼叫堆疊深處時：**

- 錯誤的值源自何處？
- 是什麼呼叫了這個帶有錯誤值的函式？
- 持續向上游追蹤直到找到源頭。
- 在「源頭」修復，而不是針對「症狀」修復。

**行動：** 使用 `search_files` 追蹤引用：

```python
# 尋找函式在哪裡被呼叫
search_files("function_name(", path="src/", file_glob="*.py")

# 尋找變數在哪裡被設定
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### 階段 1 完成檢核表

- [ ] 已完整閱讀並理解錯誤訊息。
- [ ] 問題已穩定重現。
- [ ] 已識別並審查最近的變更。
- [ ] 已收集證據（記錄、狀態、資料流）。
- [ ] 問題已隔離至特定組件/程式碼。
- [ ] 已形成根因假設 (root cause hypothesis)。

**停止：** 在你理解「為什麼」會發生之前，不要進入階段 2。

---

## 階段 2：模式分析 (Pattern Analysis)

**在修復前找出模式：**

### 1. 尋找運作正常的範例

- 在同一個程式碼庫中尋找類似且運作正常的程式碼。
- 有什麼類似的東西是運作正常的？

**行動：** 使用 `search_files` 尋找可比較的模式：

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. 與參考資料比對

- 如果正在實作某種模式，請「完整」閱讀參考實作。
- 不要略讀 — 閱讀每一行。
- 在套用前完全理解該模式。

### 3. 識別差異

- 正常運作與故障的程式碼之間有何不同？
- 列出每一個差異，不論多小。
- 不要假設「那應該沒關係」。

### 4. 理解依賴關係

- 此功能還需要哪些其他組件？
- 需要哪些設定、配置、環境？
- 它做了哪些假設？

---

## 階段 3：假設與測試 (Hypothesis and Testing)

**科學方法：**

### 1. 形成單一假設

- 清晰陳述：「我認為 X 是根因，因為 Y」。
- 寫下來。
- 要具體，不要含糊。

### 2. 進行最小化測試

- 進行「最小程度」的變更來測試假設。
- 一次只動一個變數。
- 不要同時修復多個東西。

### 3. 在繼續前進行驗證

- 有效嗎？ → 進入階段 4。
- 無效？ → 形成「新的」假設。
- 不要在此基礎上疊加更多修復。

### 4. 當你不知道時

- 誠實說出：「我不理解 X」。
- 不要假裝知道。
- 請求使用者協助。
- 進行更多研究。

---

## 階段 4：實作 (Implementation)

**修復根因，而非症狀：**

### 1. 建立失敗的測試案例

- 最簡單的重現方式。
- 盡可能使用自動化測試。
- 在修復前「必須」擁有此測試。
- 使用 `test-driven-development` 技能。

### 2. 實作單一修復

- 解決已識別的根因。
- 一次僅進行「一個」變更。
- 不要順便進行改進 (No "while I'm here")。
- 不要綁定重構 (refactoring)。

### 3. 驗證修復

```bash
# 執行特定的迴歸測試
pytest tests/test_module.py::test_regression -v

# 執行完整套件 — 確保無迴歸
pytest tests/ -q
```

### 4. 如果修復無效 — 三次法則 (The Rule of Three)

- **停止。**
- 計算：你已經嘗試了多少次修復？
- 如果 < 3 次：返回階段 1，利用新資訊重新分析。
- **如果 ≥ 3 次：停止並質疑架構 (architecture)（見下方步驟 5）。**
- 在進行架構討論前，不要嘗試第 4 次修復。

### 5. 如果 3 次以上的修復皆失敗：質疑架構

**顯示架構問題的模式：**
- 每次修復都會在不同地方揭露新的共享狀態/耦合。
- 修復需要「大規模重構」才能實作。
- 每次修復都會在其他地方產生新的症狀。

**停止並質疑基本面：**
- 這個模式從根本上來說是健全的嗎？
- 我們是否只是因為「慣性」而堅持使用它？
- 我們應該重構架構，還是繼續修復症狀？

**在嘗試更多修復前，先與使用者討論。**

這「不是」假設失敗 — 這是架構錯誤。

---

## 警訊 — 停止並遵循流程

如果你發現自己有以下想法：
- 「現在先快速修好，稍後再調查。」
- 「試試改動 X 看看會不會好。」
- 「同時加入多個變更，然後跑測試。」
- 「跳過測試，我會手動驗證。」
- 「大概是 X，讓我修修看。」
- 「我不太確定，但這可能有效。」
- 「模式說是 X，但我會用不同的方式調整。」
- 「主要問題如下：[未經調查就列出修復清單]」
- 在追蹤資料流之前就提出解決方案。
- **「再試一次修復就好」（當已經嘗試過 2 次以上時）。**
- **每次修復都在不同地方揭露新問題。**

**以上皆意味著：停止。返回階段 1。**

**如果 3 次以上修復失敗：** 質疑架構（階段 4 步驟 5）。

## 常見的合理化藉口

| 藉口 | 現實情況 |
|--------|---------|
| 「問題很簡單，不需要流程。」 | 簡單的問題也有根因。流程對於簡單的 Bug 來說執行很快。 |
| 「情況緊急，沒時間走流程。」 | 系統化偵錯比「猜測並檢查」的亂試更快。 |
| 「先試試看這個，然後再調查。」 | 第一個修復會定下模式。從一開始就做對。 |
| 「確認修復有效後我會補寫測試。」 | 未經測試的修復無法持久。先寫測試能證明修復有效。 |
| 「同時進行多個修復可以節省時間。」 | 你無法分辨是哪個修復奏效，且會導致新 Bug。 |
| 「參考資料太長，我會自己調整模式。」 | 片面的理解保證會產生 Bug。請完整閱讀。 |
| 「我看出問題了，讓我修好它。」 | 看到症狀 ≠ 理解根因。 |
| 「再試一次修復就好」（失敗 2 次以上後）。 | 3 次以上失敗 = 架構問題。質疑模式，不要再修了。 |

## 快速參考

| 階段 | 關鍵活動 | 成功準則 |
|-------|---------------|------------------|
| **1. 根因 (Root Cause)** | 閱讀錯誤、重現、檢查變更、收集證據、追蹤資料流 | 理解「什麼」以及「為什麼」 |
| **2. 模式 (Pattern)** | 尋找運行範例、比較、識別差異 | 知道哪裡不同 |
| **3. 假設 (Hypothesis)** | 形成理論、最小化測試、一次一個變數 | 確認或形成新假設 |
| **4. 實作 (Implementation)** | 建立迴歸測試、修復根因、驗證 | Bug 已解決，所有測試通過 |

## Hermes 代理整合

### 調查工具

在階段 1 使用這些 Hermes 工具：

- **`search_files`** — 尋找錯誤字串、追蹤函式呼叫、定位模式。
- **`read_file`** — 讀取包含行號的原始碼，以便進行精確分析。
- **`terminal`** — 執行測試、檢查 Git 歷史記錄、重現 Bug。
- **`web_search`/`web_extract`** — 研究錯誤訊息、函式庫文件。

### 配合 delegate_task

對於複雜的多組件偵錯，分派調查子代理：

```python
delegate_task(
    goal="調查 [特定測試/行為] 失敗的原因",
    context="""
    遵循系統化偵錯技能：
    1. 仔細閱讀錯誤訊息
    2. 重現問題
    3. 追蹤資料流以尋找根因
    4. 回報調查結果 — 先不要進行修復

    錯誤訊息：[貼上完整錯誤內容]
    檔案路徑：[指向故障程式碼的路徑]
    測試命令：[確切的命令]
    """,
    toolsets=['terminal', 'file']
)
```

### 配合 test-driven-development

修復 Bug 時：
1. 撰寫一個能重現 Bug 的測試 (RED)。
2. 系統化偵錯以尋找根因。
3. 修復根因 (GREEN)。
4. 該測試證明了修復有效並防止迴歸。

## 實際影響

來自偵錯會議的統計：
- 系統化方法：15-30 分鐘修復。
- 隨機修復方法：2-3 小時的亂試。
- 首次修復成功率：95% vs 40%。
- 引入新 Bug：趨近於零 vs 常見。

**沒有捷徑。不要猜測。系統化方法永遠勝出。**
