# 測試驅動開發 (Test-Driven Development, TDD)

## 概述

先撰寫測試。看著它失敗。撰寫最少量的程式碼使其通過。

**核心原則：** 如果你沒有親眼看到測試失敗，你就不會知道它是否測試了正確的東西。

**違反規則的字面要求，就是違反規則的核心精神。**

## 何時使用

**始終使用：**
- 新功能開發
- Bug 修復
- 重構 (Refactoring)
- 行為變更

**例外情況（需先詢問使用者）：**
- 拋棄式原型 (Throwaway prototypes)
- 自動產生的程式碼
- 配置檔案 (Configuration files)

心裡想著「就這一次跳過 TDD」？請停止。那是你在為自己找藉口。

## 鋼鐵法則 (The Iron Law)

```
先有失敗的測試，否則不准撰寫正式程式碼
```

在測試之前就寫好程式碼了？請刪掉它，重新開始。

**絕無例外：**
- 不要把它留作「參考」。
- 不要一邊寫測試一邊「調整」它。
- 不要去看它。
- 刪掉就是徹底刪除。

從測試出發重新實作。就這麼簡單。

## RED-GREEN-REFACTOR 循環

### RED — 撰寫失敗的測試

撰寫一個最簡單的測試，說明預期應該發生的行為。

**好的測試：**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
名稱清晰、測試實際行為、專注於一件事。

**壞的測試：**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # 沒測到重試次數？時序？
```
名稱模糊、測試的是 Mock 而非實際程式碼。

**要求：**
- 每個測試僅包含一個行為。
- 名稱清晰具描述性（名稱中有 "and"？請拆分成兩個測試）。
- 使用實際程式碼，而非 Mock（除非真的避無可避）。
- 名稱描述的是「行為」，而非「實作方式」。

### 驗證 RED — 看著它失敗

**強制執行，絕不可跳過。**

```bash
# 使用 terminal 工具執行特定測試
pytest tests/test_feature.py::test_specific_behavior -v
```

確認：
- 測試確實失敗（而非因打錯字產生的錯誤）。
- 失敗訊息符合預期。
- 失敗原因是因為「缺少該功能」。

**測試直接通過了？** 代表你在測試現有的行為。請修正測試。

**測試報錯 (Error)？** 修正錯誤，重新執行直至其「正確地失敗」。

### GREEN — 撰寫最少量的程式碼

撰寫能讓測試通過的最簡單程式碼。不多不少。

**正確做法：**
```python
def add(a, b):
    return a + b  # 不加額外內容
```

**錯誤做法：**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # 這是多餘的！
    return result
```

不要新增功能、不要重構其他程式碼，也不要進行測試範圍之外的「改進」。

**在 GREEN 階段可以「走捷徑」：**
- 硬編碼 (Hardcode) 傳回值
- 複製貼上
- 重複程式碼
- 跳過邊界情況

我們會在 REFACTOR 階段處理這些問題。

### 驗證 GREEN — 看著它通過

**強制執行。**

```bash
# 執行特定測試
pytest tests/test_feature.py::test_specific_behavior -v

# 接著執行「所有」測試以確保無迴歸
pytest tests/ -q
```

確認：
- 測試通過。
- 其他測試依然通過。
- 輸出乾淨（無錯誤、無警告）。

**測試失敗？** 修正程式碼，而非測試。

**其他測試失敗？** 立即修復迴歸 (regressions) 問題。

### REFACTOR — 清理程式碼

僅在通過 GREEN 階段後進行：
- 移除重複內容。
- 改進命名。
- 擷取輔助函式。
- 簡化運算式。

過程中保持測試皆為通過狀態 (Green)。不要加入新行為。

**如果重構期間測試失敗：** 立即還原，並改採更小的步驟進行。

### 重複上述步驟

為下一個行為撰寫下一個失敗的測試。一次只跑一個循環。

## 為何順序至關重要

**「我會先寫程式碼，之後再寫測試來驗證。」**

在程式碼之後撰寫的測試會「立即通過」。立即通過無法證明任何事：
- 可能測錯了東西。
- 可能測的是實作細節，而非行為。
- 可能遺漏了你忘記的邊界情況。
- 你從未看過它捕捉到 Bug。

「測試先行」強迫你看到測試失敗，證明它確實測到了某些東西。

**「我已經手動測試過所有邊界情況了。」**

手動測試是臨時且不系統的。你以為測過了一切，但：
- 沒有測試紀錄。
- 程式碼變更時無法重複執行。
- 在壓力下容易遺漏案例。
- 「我試的時候是好的」不等於全面性。

自動化測試是系統化的，每次執行的路徑都相同。

**「刪掉 X 小時的工作太浪費了。」**

這是「沉沒成本謬誤」。時間已經花掉了。你現在的選擇是：
- 刪除並使用 TDD 重新實作（高信心）。
- 保留它並在之後補測試（低信心，極可能有 Bug）。

真正的「浪費」是保留你無法信任的程式碼。

**「TDD 太教條化了，務實的做法應該靈活調整。」**

TDD 「就是」務實的做法：
- 在 commit 前發現 Bug（比之後偵錯更快）。
- 防止迴歸（測試能立即攔截錯誤）。
- 文件化行為（測試展示了如何使用程式碼）。
- 賦予重構能力（自由修改，測試會幫你把關）。

「務實」的捷徑 = 在正式環境偵錯 = 速度更慢。

**「事後補測試也能達到同樣目標 — 重點在精神而非形式。」**

不對。事後補測試是在回答「這段程式碼做了什麼？」，測試先行則是在回答「這段程式碼『應該』做什麼？」。

事後補測試會受到你實作方式的偏見影響。你測的是你「做出來的東西」，而非「要求的東西」。測試先行則強迫你在實作前發現邊界情況。

## 常見的合理化藉口

| 藉口 | 現實情況 |
|--------|---------|
| 「這太簡單了，不用測。」 | 簡單的程式碼也會出錯。寫測試只需 30 秒。 |
| 「我晚點再補測試。」 | 立即通過的測試無法證明任何事。 |
| 「事後補測試也一樣。」 | 事後補 =「這做了什麼？」；測試先行 =「這應該做什麼？」 |
| 「我已經手動測過了。」 | 臨時測試 ≠ 系統化。無紀錄、無法重複執行。 |
| 「刪掉 X 小時工作太浪費。」 | 沉沒成本謬誤。保留未經驗證的程式碼是技術債。 |
| 「留著參考，我會先寫測試。」 | 你一定會忍不住去套用它。那本質上就是事後補測試。刪掉就是徹底刪除。 |
| 「我需要先探索一下。」 | 沒問題。請把探索用的程式碼丟掉，重新以 TDD 開始。 |
| 「測試很難寫 = 設計不清晰。」 | 請傾聽測試的回饋。難測 = 難用。 |
| 「TDD 會拖慢我的速度。」 | TDD 比偵錯更快。務實 = 測試先行。 |
| 「手動測試比較快。」 | 手動無法證明邊界案例。每次變動你都得重新手動測一遍。 |
| 「現有程式碼沒測試。」 | 你正在改進它。請為你動到的程式碼加上測試。 |

## 警訊 — 停止並重新開始

如果你發現自己在做以下任何事，請刪除程式碼並以 TDD 重新開始：

- 先寫程式碼，再寫測試。
- 實作完成後才寫測試。
- 測試第一次執行就直接通過。
- 無法解釋為什麼測試失敗。
- 測試是「之後」才補上的。
- 找藉口說「就這一次」。
- 「我已經手動測過了」。
- 「事後補測試也能達到同樣目的」。
- 「留著參考」或「調整現有的程式碼」。
- 「已經花了 X 小時，刪掉太浪費」。
- 「TDD 太教條，我是在追求務實」。
- 「這次情況不同，因為……」。

**以上皆意味著：刪除程式碼，以 TDD 重新開始。**

## 驗證檢核表

在標記工作完成之前：

- [ ] 每個新函式/方法都有對應的測試。
- [ ] 在實作前，親眼看到每個測試失敗。
- [ ] 每個測試都因預期原因失敗（功能缺失，而非打錯字）。
- [ ] 撰寫了最少量的程式碼使測試通過。
- [ ] 所有測試皆通過。
- [ ] 輸出乾淨（無錯誤、無警告）。
- [ ] 測試使用實際程式碼（除非避無可避才用 Mock）。
- [ ] 已涵蓋邊界情況與錯誤處理。

無法勾選所有項目？代表你跳過了 TDD。請重新開始。

## 遇到瓶頸時

| 問題 | 解決方案 |
|---------|----------|
| 不知道該怎麼測 | 撰寫你理想中的 API。先寫斷言 (Assertion)。詢問使用者。 |
| 測試太複雜 | 代表設計太複雜。請簡化介面。 |
| 必須對所有東西用 Mock | 代表程式碼耦合度太高。請使用依賴注入 (Dependency Injection)。 |
| 測試設定 (Setup) 非常龐大 | 擷取輔助函式。依然複雜？請簡化設計。 |

## Hermes 代理整合

### 執行測試

在每個步驟中使用 `terminal` 工具執行測試：

```python
# RED — 驗證失敗
terminal("pytest tests/test_feature.py::test_name -v")

# GREEN — 驗證通過
terminal("pytest tests/test_feature.py::test_name -v")

# 完整套件 — 驗證無迴歸
terminal("pytest tests/ -q")
```

### 配合 delegate_task

分派子代理進行實作時，在目標中強制要求 TDD：

```python
delegate_task(
    goal="使用嚴格的 TDD 實作 [功能名稱]",
    context="""
    遵循測試驅動開發 (test-driven-development) 技能：
    1. 「先」撰寫失敗的測試
    2. 執行測試並驗證其失敗
    3. 撰寫最少量的程式碼使其通過
    4. 執行測試並驗證其通過
    5. 如有需要，進行重構
    6. Commit

    專案測試命令：pytest tests/ -q
    專案結構：[描述相關檔案]
    """,
    toolsets=['terminal', 'file']
)
```

### 配合 systematic-debugging

發現 Bug 了？撰寫一個能重現該 Bug 的失敗測試。遵循 TDD 循環。該測試能證明修復有效，並防止未來再次發生。

絕對不要在沒有測試的情況下修復 Bug。

## 測試反模式 (Anti-Patterns)

- **測試 Mock 行為而非實際行為** — Mock 應該用來驗證互動，而非取代被測系統。
- **測試實作細節** — 測試應針對行為/結果，而非內部方法的呼叫過程。
- **只測成功路徑 (Happy path)** — 務必測試邊界情況、錯誤處理與極限。
- **脆弱的測試** — 測試應驗證行為而非結構；重構不應導致測試失敗。

## 最終準則

```
正式程式碼 → 測試已存在且先失敗過
否則 → 不是 TDD
```

除非使用者明確許可，否則絕無例外。
