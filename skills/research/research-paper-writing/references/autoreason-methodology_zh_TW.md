# Autoreason：迭代改進方法論 (Iterative Refinement Methodology)

這是 autoreason 迭代改進方法的完整參考，源自於對主觀寫作任務、競爭性程式設計以及四個模型層級的實驗結果。當任何輸出（論文草稿、實驗腳本、分析、任務定義）需要進行迭代改進時，請使用此方法。

**來源**：[NousResearch/autoreason](https://github.com/NousResearch/autoreason) — "Autoreason: When Iterative LLM Refinement Works and Why It Fails"

---

## 策略選擇指南

### 決策樹

```
任務是否具備客觀可驗證性（程式碼、數學、事實）？
├── 是 → 模型是否能在第一次嘗試時解決？
│   ├── 是 → 使用單次生成 (single pass)（不需要改進）
│   └── 否 → 使用 autoreason（結構化分析 → 基於推理的修訂）
│
└── 否（主觀性任務） → 您使用的是哪個模型層級？
    ├── 弱 (Weak) (Llama 8B, 小型模型)
    │   → 單次生成。模型太弱，改進無法帶來幫助。
    │     應投資於生成品質，而非迭代。
    │
    ├── 中階 (Mid-tier) (Haiku 3.5, Gemini Flash)
    │   → 搭配較強審查員的 Autoreason。這是最佳切入點。
    │     自我修訂會毀掉弱模型的輸出 — autoreason 可防止此情況。
    │
    ├── 強 (Strong) (Sonnet 4)
    │   → 針對開放式任務使用 Autoreason。勝率 3/5。
    │     針對具體的技術任務使用「批評與修訂 (Critique-and-revise)」(2/5)。
    │
    └── 頂尖 (Frontier) (Sonnet 4.6, Opus)
        ├── 範圍受限？ → Autoreason。在受限任務中勝率 2/3。
        └── 未受限？ → 批評與修訂或單次生成。
            Autoreason 在未受限的頂尖模型任務中會失敗（排在最後）。
```

### 策略比較表

| 策略 | 最適合 | 應避免的情況 | 運算量 (每輪迭代) |
|----------|----------|------------|------------------------|
| **單次生成 (Single pass)** | 頂尖模型、模板任務、預算緊張 | 品質上限較低的中階模型 | 1 次呼叫 |
| **批評與修訂 (Critique-and-revise)** | 具體技術要求（系統設計、規格說明） | 弱模型（品質下降）、未受限的主觀任務 | 2 次呼叫 |
| **Autoreason** | 中階模型、受限範圍、具有真實權衡 (tradeoffs) 的任務 | 弱模型 (Llama 8B)、頂尖模型 + 未受限任務 | ~6 次呼叫 |
| **Best-of-N** | 幾乎不建議使用 | 特別是弱模型 — 效果比單次生成還差 | N 次呼叫 |

### 為何各策略會失敗

| 策略 | 失敗模式 | 機制 |
|----------|-------------|-----------|
| **單次生成** | 品質上限 | 沒有超越第一次嘗試的改進機制 |
| **批評與修訂** | 逐漸退化 (Progressive degradation) | 模型幻覺出問題（趨炎附勢 sycophancy）、每輪範圍蔓延、從不拒絕修改 |
| **Best-of-N** | 隨機選擇 | 缺乏良好的排名信號時，樣本越多 = 平庸選項越多 |
| **Autoreason (未受限)** | 綜合漂移 (Synthesis drift) | 強大模型產生的「綜合版 (syntheses)」被偏好的程度非常一致，導致「現任版 (incumbent)」永不穩定 |

## Autoreason 迴圈 (The Autoreason Loop)

### 架構

```
┌──────────────────────────────────────────────────────────┐
│                    ITERATION LOOP                         │
│                                                           │
│   現任版 A ──► 批評者 ──► 作者 B ──► 綜合者     │
│       │                                      │            │
│       │              ┌───────────────────────┘            │
│       ▼              ▼                                    │
│      [A]           [AB]          [B]                      │
│       │              │            │                       │
│       └──────────────┼────────────┘                       │
│                      ▼                                    │
│              評審小組 (雙盲)                          │
│                      │                                    │
│                      ▼                                    │
│                   勝出者                                  │
│                      │                                    │
│              ┌───────┴───────┐                            │
│              ▼               ▼                            │
│         A 連續勝出 k=2 次   B 或 AB 勝出                  │
│         收斂？              → 成為新的現任版              │
│              │                                            │
│              ▼                                            │
│           已收斂 (CONVERGED)                              │
└──────────────────────────────────────────────────────────┘
```

### 角色 (Roles)

每個角色都是**全新的、隔離的代理 (fresh, isolated agent)**，沒有共享的上下文：

| 角色 | 輸入 | 輸出 | 核心規則 |
|------|-------|--------|----------|
| **批評者 (Critic)** | 任務 + 現任版 A | 問題列表 | 僅找出問題。不提供修正。不提供建議。 |
| **作者 B (Author B)** | 任務 + A + 批評 | 修訂版 B | 針對每項批評進行修改。說明每項更動修正了哪個問題。 |
| **綜合者 (Synthesizer)** | 任務 + X + Y (標籤隨機化) | 綜合版 AB | 擷取兩者的最強項。不是折衷方案。 |
| **評審小組 (Judge Panel)** | 任務 + A, AB, B (標籤與順序隨機化) | 排名 | 從最好到最差進行排名。不具備作者立場。 |

### 設定 (Configuration)

| 參數 | 數值 | 理由 |
|-----------|-------|-----------|
| **收斂閾值 k** | 2 | k=1 太早（94% 的情況隨後會被取代）。k=2 則 100% 收斂，品質趨於平穩。k=3 失敗率 24%，成本 2 倍且無品質提升。 |
| **作者溫度 (Temperature)** | 0.7-0.8 | 鼓勵多樣化的修訂 |
| **評審溫度 (Temperature)** | 0.3 | 鼓勵一致的評估 |
| **迴圈內評審** | 3 | 平衡每輪成本與評估穩定性 |
| **最終評估評審** | 7 | 針對最終比較提供更高的統計檢定力 (statistical power) |
| **最大標記 (Max tokens)** | 4096 | 標準；長篇內容（論文）使用 8192 |
| **評審類型** | 思維鏈 (Chain-of-thought) | 在某些任務中收斂速度快 3 倍。務必使用。 |
| **平手處理** | 保守（現任版勝出） | 防止偽陽性 (false positives) — A 必須被真實擊敗 |
| **最大輪數** | 25 (受限), 50 (補救) | 安全上限；大多數在 10-15 輪內收斂 |

### 提示詞 (Prompts)

#### 批評者 (Critic)
```
System: 您是一位嚴厲的審查員。您的唯一任務是找出真實的問題。
請具體且具象。不要建議修正方案。

User: 請找出這份提案的真實問題。重點關注：
- 描述中無法運作的部分
- 益處不足以抵消其複雜度的部分
- 錯誤的假設
- 遺漏的部分
不要提出修正方案。只需指出問題。
```

#### 作者 B (Author B)
```
System: 您是一位資深顧問，正根據具體的批評意見修訂一份提案。直接解決每項有效的批評。不要進行與已識別問題無關的更動。

User: [TASK] + [VERSION A] + [CRITIC OUTPUT]
請進行修訂以解決這些問題。針對每項更動，請說明它修正了哪個問題。
```

#### 綜合者 (Synthesizer)
```
System: 您會收到兩個版本作為平等的輸入。請從兩者中擷取最強的元素，並產生一個連貫的綜合版本。這不是一個折衷方案。

User: [TASK] + [VERSION X] + [VERSION Y]
（標籤隨機化 — 綜合者不知道哪個是現任版）
```

#### 評審 (思維鏈 CoT) — 務必使用此版本
```
System: 您是一位獨立的評估者。在做出決定前請仔細思考。

User: [TASK] + 三份提案。針對每一份，請逐步思考：
1. 它哪裡做得對？
2. 它哪裡做錯或遺漏了？
3. 數據和聲明是否合理 (defensible)？
4. 細節是適當還是臃腫？
經過推理後，對這三份進行排名。
RANKING: [最好], [第二], [最差]
```

#### 基準提示詞 (用於比較實驗)

| 基準 | 提示詞 |
|----------|--------|
| **保守 (Conservative)** | "在保留有效部分的同時進行最小幅度的改進。不要增加新章節或顯著擴大範圍。" |
| **改進此內容 (Improve this)** | "改進這份文件。"（無進一步指導） |
| **嚴厲批評 (Harsh critic)** | "進行批判性評估並重寫，修正您識別出的所有弱點。" |
| **批評與修訂 (Critique & revise)** | 步驟 1："產出結構化的批評，列出具體的弱點。" 步驟 2："進行修訂以解決每項批評。" |

## 評分：波達計數法 (Borda Count)

評審對候選版本進行排名。根據排名位置授予分數：

| 排名 | 分數 (針對 3 個候選者) |
|------|----------------------|
| 第一名 | 3 |
| 第二名 | 2 |
| 第三名 | 1 |

**彙整**：加總所有評審的分數。勝出者 = 總分最高者。
**平手處理**：若分數相同，由現任版 (A) 勝出。

**範例** (3 位評審)：
- 評審 1: AB > A > B → AB 得 3 分, A 得 2 分, B 得 1 分
- 評審 2: A > AB > B → A 得 3 分, AB 得 2 分, B 得 1 分
- 評審 3: AB > B > A → AB 得 3 分, B 得 2 分, A 得 1 分
- 總分：AB=8, A=6, B=4 → AB 勝出，成為新的現任版

**每位評審的隨機化**：
- 候選標籤隨機化（對一位評審來說 A 可能叫「提案 X」，對另一位則叫「提案 Z」）
- 呈現順序隨機化（AB 可能出現在第一個或最後一個）
- 這樣可以防止位置偏誤和標籤偏誤

## 模型選擇指南

### 各模型層級的實證結果

| 模型 | Autoreason 勝場 | Autoreason 平均 Borda | 最佳基準 | 差距 | 建議 |
|-------|----------------|---------------------|---------------|--------|----------------|
| **Llama 3.1 8B** | 1/3 | 23.7 | 25.0 (單次) | -1.3 | 跳過 autoreason。模型太弱，無法產生多樣化的候選者。 |
| **Gemini 2.0 Flash** | 2/3 | 25.0 | 20.0 (單次) | +5.0 | 不錯的候選對象。有適度增益。 |
| **Haiku 3.5** | 3/3 | **42.0** | 33.7 (單次) | **+8.3** | **最佳候選對象。** 滿分表現。基準方法會主動破壞品質。 |
| **Sonnet 4** | 3/5 | 27.8 | 22.4 (C&R) | +5.4 | 開放式任務的良好選擇。技術任務使用 C&R 較佳。 |
| **Sonnet 4.6 (未受限)** | 0/1 | 7.0 | 31.0 (C&R) | -24.0 | 不要針對未受限任務使用 autoreason。 |
| **Sonnet 4.6 (受限)** | 2/3 | 29.0 | 27.0 (改進) | +2.0 | 僅在有範圍限制時使用。 |

### 生成-評估差距 (The Generation-Evaluation Gap)

核心洞察：**autoreason 的價值取決於模型「生成能力」與「自我評估能力」之間的差距。**

```
弱模型 (Llama 8B)：
  生成：差  |  自我評估：差
  差距：小（兩者皆差） → Autoreason 無法提供幫助，無多樣化的候選者

中階模型 (Haiku, Flash)：
  生成：尚可  |  自我評估：差
  差距：大 → Autoreason 的甜蜜點。外部評估彌補了這個差距。

強模型 (Sonnet 4)：
  生成：好  |  自我評估：尚可
  差距：適中 → Autoreason 在 3/5 的任務中提供幫助

頂尖模型 (Sonnet 4.6)：
  生成：極佳  |  自我評估：好
  差距：小 → 簡單的方法即可勝任。Autoreason 在未受限任務中反而有害。
```

**實務規則**：隨著模型成本下降與能力提升，今日的頂尖模型將成為明日的中階模型。「生成-評估差距」是結構性的，而非暫時的。請根據模型在能力曲線上的位置來匹配改進架構。

### 評審選擇

| 作者模型 | 建議評審 | 理由 |
|-------------|------------------|-----------|
| Llama 8B | 不要使用 autoreason | 模型太弱 |
| Gemini Flash | Sonnet 4 | 跨模型評估行之有效 |
| Haiku 3.5 | Sonnet 4 | 強大的外部評估是其機制所在 |
| Haiku 3.5 | Haiku 3.5 (相同) | 仍然有效 — 錦標賽結構即使沒有強大評審也能提供價值（平均 Borda 20.7 vs 18.3） |
| Sonnet 4 | Sonnet 4 (相同) | 此層級的同模型評審有效 |
| Sonnet 4.6 | Sonnet 4.6 (相同) | 僅限於有範圍限制時 |

## 範圍限制設計 (Scope Constraint Design)

### 是什麼讓 Autoreason 在受限任務中奏效

同一個模型 (Sonnet 4.6) 透過範圍限制，能從**最後一名**（未受限）躍升至**第一名**（受限）。限制劃定了改進空間的邊界，使「綜合版漂移 (synthesis drift)」無法累積。

### 有效的限制

| 限制類型 | 範例 | 為何有效 |
|----------------|---------|-------------|
| **固定事實** | "僅使用這 8 個數據點，不添加任何其他內容" | 限制了資訊空間 |
| **固定交付物** | "500 字的創業推廣 (pitch)" (不是 "改進此內容") | 定義了完成條件 |
| **固定結構** | "精確包含 4 個章節，每個章節有 3 個編號項目" | 防止結構性漂移 |
| **固定修改項** | "精確解決這 3 個審查員的疑慮" | 劃定了修改範圍的邊界 |

### 無效的限制

| 限制 | 為何失敗 | 會發生什麼事 |
|-----------|-------------|-------------|
| 僅限字數 | 不是範圍限制 | 偽收斂 (False convergence) — 因為長度而非品質被拒絕 |
| "請簡潔" | 太籠統 | 2-3 輪後就被忽視 |
| "請詳盡" | 反向限制 | 招致範圍蔓延 |
| 完全沒有限制 | 無邊界的改進空間 | 綜合版佔據主導，無法收斂 |

### 任務類別

| 任務類型 | Autoreason 是否奏效？ | 理由 |
|-----------|-------------------|-----|
| 具有真實權衡的任務（策略、政策） | 是 | 具備多個有效方案供錦標賽選擇 |
| 受限寫作（推廣、備忘錄、檢討報告） | 大部分 (2/3) | 範圍受限，評估準則明確 |
| 模板填充（事故檢討報告） | 否 | 結構唯一正確，決策空間最小 |
| 競爭性程式設計 | 是 | 自然受限，測試套件提供外部驗證 |
| 開放式未受限 + 頂尖模型 | 否 | 綜合版漂移，無法收斂 |

## 失敗分類學 (Failure Taxonomy)

| 失敗模式 | 條件 | 偵測方法 | 證據 |
|-------------|-----------|-----------|----------|
| **自我修正不可靠** | 缺乏外部評估信號 | 基準方法退化至單次生成以下 | Haiku 基準：平均 16.3 vs 單次 33.7 |
| **漂移 / 綜合版主導** | 未受限的範圍 | A 勝率 <15%，AB 佔主導 | Sonnet 4.6 未受限：A 勝 12%，AB 勝 60%+ |
| **過度擬合可見反饋** | 淺層修訂迴圈 (C&R) | 公開/私有測試發散嚴重 | C&R 在困難程式題上過度擬合 32% |
| **無法收斂** | 評審流程損壞 | 解析失敗，有效評審 <3 位 | 混合評審小組解析失敗：11 輪以上 |
| **模型太弱** | 生成多樣性不足 | 所有候選版本看起來都一樣 | Llama 8B 僅在 1/3 的任務中勝出 |

### 補救模式

| 失敗 | 補救措施 |
|---------|----------|
| 無法收斂 (漂移) | 對任務增加範圍限制 |
| 無法收斂 (評審損壞) | 修正解析器，確保繼續前有 3 位有效評審 |
| 品質隨迭代退化 | 切換回單次生成或增加限制 |
| 模型太弱 | 使用較強模型進行生成，弱模型負責低成本角色 |
| 過度擬合 (程式碼) | 使用結構化分析步驟，而非僅依賴測試反饋 |

## 程式碼領域改編

Autoreason 方法在程式碼與寫作領域的應用有所不同：

### 寫作領域
```
呼叫 1：批評者 (找出現任版的問題)
呼叫 2：作者 B (根據批評進行修訂)
呼叫 3：綜合者 (合併 A 與 B)
呼叫 4-6：評審小組 (3 位雙盲評審對 A, B, AB 進行排名)
```

### 程式碼領域 (6 次呼叫預算)
```
呼叫 1：初始生成
呼叫 2：結構化分析（5 個要點 — 禁止寫程式）：
  - 問題分析：問題真正的要求是什麼？
  - 方法分析：我們使用了什麼方法，是否正確？
  - 失敗分析：為什麼測試失敗了？
  - 替代方案：還有什麼其他可行的方法？
  - 邊際案例：哪些輸入可能會破壞解決方案？
呼叫 3-6：基於推理的修訂
  - 每次修訂都必須解釋「為什麼」它能解決問題
  - 可看到公開（可見）測試案例的測試結果
```

**關鍵差異**：程式碼策略將評審小組替換為測試套件評估（客觀事實）。「結構化分析」步驟（呼叫 2）是推動復原的關鍵 — 它在嘗試修正前，強迫對「為何方法失敗」進行推理。

**結果**：復原是其機制所在。在 autoreason 與單次生成初始皆失敗的題目中，autoreason 的復原率為 62%，高於單次生成的 43% (McNemar p=0.041, Cohen's h=0.32)。

## 將 Autoreason 應用於論文寫作

該論文本身就是使用 autoreason 進行精煉的（論文第 8 節）：

### 設定
- 模型：claude-opus-4
- 評審：3 位 Opus 評審
- 增強：地面事實批評者 (Ground-truth critic)（可存取實際實驗數據）
- 結果：在 9 輪內收斂

### 論文精煉的關鍵發現

1. **地面事實批評者至關重要**：若無地面事實存取權，Opus 會幻覺出捏造的消融實驗、錯誤的信賴區間、錯誤的模型名稱以及錯誤的角色描述。有了地面事實存取權，批評者在第一輪就抓住了所有四個問題。

2. **評審小組的完整性很重要**：一位評審的解析器損壞（Gemini 輸出格式不匹配），導致小組從 3 位減少到 2 位評審。這使得 11 輪以上都無法收斂。修正為 3 位正常運作的評審後，同樣的現任版在 2 輪內就收斂了。損壞的評審不會只增加雜訊 — 它會阻止平衡。

### 建議的論文精煉設定

```
批評者提示詞："您正在審閱一份研究論文草稿。您可以存取實際的實驗結果 [GROUND TRUTH DATA]。請找出事實錯誤、缺乏支持的聲明、幻覺出的結果以及結構性問題。不要建議修正方案。"
```

## 運算預算參考

| 方法 | 每輪呼叫次數 | 典型輪數 | 總呼叫次數 | 相對成本 |
|--------|---------------|----------------|-------------|---------------|
| 單次生成 | 1 | 1 | 1 | 1x |
| Best-of-N | N | 1 | N | Nx |
| 批評與修訂 | 2 | 15 | 30 | 30x |
| Autoreason (迴圈內) | ~6 | 10-15 | 60-90 | 60-90x |
| Autoreason (含最終評估) | ~6 + 7 | 10-15 + 1 | 67-97 | ~80x |

**成本-品質權衡**：Autoreason 每輪使用的運算量約高出 6 倍，且通常執行更多輪。這是一個真實的權衡。該方法以運算量換取評估品質。在受限任務與中階模型上，這種權衡是非常正向的。在未受限任務與頂尖模型上，則為負向。

**CoT 評審可降低成本**：1 位 CoT 評審提供的評估品質與 3 位標準評審相當，且可節省約 40% 的成本。請務必使用 CoT 評審。
