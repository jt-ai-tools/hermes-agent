## 敘事原則

### 來自 Neel Nanda 的建議

「一篇論文是一個簡短、嚴謹、基於證據的技術故事，並帶有讀者關心的結論。」

敘事建立在三大支柱之上，這些支柱必須在緒論結束前清晰呈現：

**「是什麼 (What)」**：一到三個符合連貫主題的具體新穎主張。模糊的貢獻（如「我們研究了 X」）會立即失敗——審稿人需要精確、可證偽的主張。

**「為什麼 (Why)」**：嚴謹的實證證據，能令人信服地支持這些主張，包括誠實調優的強大基準測試 (Baselines)，以及能區分競爭假設的實驗，而不僅僅是展示「不錯的結果」。

**「所以呢 (So What)」**：讀者為什麼要關心？將你的貢獻與社群公認的重要問題聯繫起來。

### 來自 Andrej Karpathy 的建議

「論文不是你報告的實驗隨機集合。論文是在推銷一件之前不明顯或不存在的單一事物。整篇論文都以手術般的精確度圍繞著這個核心貢獻進行組織。」

這適用於你提出新架構、理論結果或對現有方法改進的理解——NeurIPS 明確指出「原創性並不一定需要全新的方法」。

**實際影響**：如果你不能用一句話陳述你的貢獻，那你還沒有寫出一篇論文。其他一切——實驗、相關工作、討論——都只是為了支持那個核心主張而存在。

## 時間分配

### 來自 Neel Nanda 的建議

在以下每一項投入**大約相同的時間**：
1. 摘要 (Abstract)
2. 緒論 (Introduction)
3. 圖表 (Figures)
4. 其他所有內容的總和

這並非誇張——大多數審稿人在閱讀你的方法論章節之前就已經形成了初步判斷。讀者接觸你論文的模式是可以預測的：**標題 → 摘要 → 緒論 → 圖表 → 也許是其餘部分。**

### 審稿人的閱讀模式

對審稿人行為的研究顯示：
- 摘要的閱讀率是 100%
- 90% 以上的審稿人會瀏覽緒論
- 大多數審稿人在查看方法論之前會先查看圖表
- 只有在建立興趣後才會閱讀完整的法論

**啟示**：將論文的價值前置。不要埋沒你的貢獻。

## 摘要寫作公式

### Sebastian Farquhar 的五句公式

1. **你取得了什麼成就**：「我們引入了……」、「我們證明了……」、「我們展示了……」
2. **為什麼這很困難且重要**
3. **你是如何做到的**（使用專業關鍵字以利搜尋）
4. **你有什麼證據**
5. **你最顯著的數據/結果**

### 範例（優質摘要）

```
我們證明了在過度參數化的神經網路上的梯度下降，
會以線性速率收斂到全域最小值。[成就]
這解決了一個關於為什麼深度學習儘管存在非凸優化景觀
卻仍能運行的基本問題。[困難/重要性]
我們的證明依賴於展示神經切線核 (Neural Tangent Kernel) 在訓練期間
保持近似常數，從而將問題簡化為核回歸 (Kernel Regression)。[如何做到的，含關鍵字]
我們在 CIFAR-10 和 ImageNet 上驗證了我們的理論，結果顯示
預測的收斂速率與實驗結果誤差在 5% 以內。[證據]
這是第一個針對實際深度和寬度網路的
多項式時間收斂保證。[顯著結果]
```

### 應避免的事項

來自 Zachary Lipton 的建議：「如果第一句話可以放在任何一篇 ML 論文之前，那就刪掉它。」

**刪掉這些開場白**：
- 「大型語言模型取得了巨大的成功……」
- 「深度學習徹底改變了……」
- 「近年來，神經網路已經……」

**直接從你的具體貢獻開始。**

## 緒論結構

### 要求

- **最多 1-1.5 頁**（雙欄格式）
- **方法論應從第 2-3 頁開始**
- 必須包含 **2-4 個點列式的貢獻清單**（每項最多 1-2 行）

### 結構模板

```markdown
1. 開場勾子 (Opening Hook) (2-3 句)
   - 陳述論文要解決的問題
   - 為什麼現在這個問題很重要

2. 背景/挑戰 (1 段)
   - 是什麼讓這個問題變得困難？
   - 其他人嘗試過什麼？為什麼不足夠？

3. 你的方法 (1 段)
   - 你有什麼不同的做法？
   - 使你的貢獻成為可能的關鍵洞察

4. 貢獻要點 (2-4 項)
   - 要具體且可證偽
   - 每個要點：最多 1-2 行

5. 結果預覽 (2-3 句)
   - 最令人印象深刻的數據
   - 評估範圍

6. 論文架構（可選，1-2 句）
   - 「第 2 節介紹……第 3 節描述……」
```

### 貢獻要點：好與壞的對比

**好：**
- 我們證明了在假設 Y 下，X 在 O(n log n) 時間內收斂
- 我們引入了 Z，一種可減少 40% 記憶體的三層架構
- 我們展示了在基準測試 C 上，A 的表現優於 B 15%

**壞：**
- 我們研究了 X 的問題（這不是貢獻）
- 我們提供了廣泛的實驗（太過模糊）
- 我們對該領域做出了幾項貢獻（什麼都沒說）

## 句子層級的清晰度

### 來自 Gopen & Swan：《科學寫作的科學》

George Gopen 和 Judith Swan 在 1990 年發表的開創性論文中確立了：**讀者對資訊出現在散文中的位置有結構性期望**。違反這些期望會迫使讀者將精力花在結構而非內容上。

> 「如果讀者要掌握作者的意思，作者必須理解讀者的需求。」

#### 讀者期望的七大原則

**原則 1：主詞與動詞鄰近性 (Subject-Verb Proximity)**

保持語法上的主詞和動詞靠近。任何插入的內容都會被視為次要的重要性干擾。

**弱**：「該模型，它在 1 億個標記上進行訓練，並使用秩為 16 的 LoRA 在特定領域數據上進行微調，取得了最先進的結果。」
**強**：「該模型在 1 億個標記上進行訓練，並使用 LoRA（秩為 16）微調後，取得了最先進的結果。」

**原則 2：強調位置（把最好的留到最後）(Stress Position)**

讀者會自然地強調**句子的最後幾個字**。將最重要的資訊放在那裡。

**弱**：「使用注意力機制時，準確度提高了 15%。」
**強**：「當使用注意力機制時，準確度提高了 **15%**。」

**原則 3：主題位置（重要的事情先說）(Topic Position)**

句子的開頭建立了視角。將「是誰的故事」的元素放在第一位——讀者期望句子是關於首先出現的人或物。

**弱**：「一種計算對齊分數的新穎注意力機制被引入了。」
**強**：「為了體解決對齊問題，我們引入了一種新穎的注意力機制。」

**原則 4：舊資訊先於新資訊 (Old Information Before New)**

將熟悉的資訊（舊資訊）放在主題位置以建立向後連結；將新資訊放在強調位置以示強調。

**弱**：「稀疏注意力是由 Child 等人引入的。標準注意力的二次複雜度激發了這項工作。」
**強**：「標準注意力具有二次複雜度。為了體解決這個問題，Child 等人引入了稀疏注意力。」

**原則 5：一個單元，一個功能 (One Unit, One Function)**

每個論述單元（句子、段落、章節）應該只服務一個功能。如果你有兩個重點，請使用兩個單元。

**原則 6：動詞表達行動 (Articulate Action in the Verb)**

在動詞中表達每個句子的行動，而不是在名詞化的名詞中。

**弱**：「我們對結果進行了分析 (performed an analysis)。」（名詞化）
**強**：「我們分析了 (analyzed) 結果。」（行動在動詞中）

**原則 7：新資訊前的背景 (Context Before New Information)**

在要求讀者考慮任何新事物之前提供背景。這適用於所有層級——句子、段落、章節。

**弱**：「等式 3 顯示當學習率滿足……時，收斂得到保證。」
**強**：「為了保證收斂，學習率必須滿足等式 3 中的條件……」

#### 總結表

| 原則 | 規則 | 記憶法 |
|-----------|------|----------|
| 主詞與動詞鄰近性 | 保持主詞和動詞靠近 | 「不要打斷你自己」 |
| 強調位置 | 在句尾強調 | 「把最好的留到最後」 |
| 主題位置 | 在句首提供背景 | 「重要的事情先說」 |
| 舊資訊先於新資訊 | 熟悉 → 不熟悉 | 「建立在已知基礎上」 |
| 一個單元，一個功能 | 每個段落 = 一個觀點 | 「一個容器裝一個想法」 |
| 動詞表達行動 | 使用動詞，而非名詞化 | 「動詞做事，名詞坐著」 |
| 新資訊前的背景 | 先解釋再呈現 | 「先搭好舞台」 |

## 微觀層級的寫作技巧

### 來自 Ethan Perez (Anthropic) 的建議

這些實用的微觀技巧能提高句子和單字層級的清晰度。

#### 代名詞管理

**盡量減少代名詞**（「這 (this)」、「它 (it)」、「這些 (these)」、「那 (that)」）。當代名詞是必要時，請將它們作為帶有名詞的形容詞使用：

**弱**：「這顯示模型收斂了。」
**強**：「這個結果顯示模型收斂了。」

**弱**：「它提高了性能。」
**強**：「這項修改提高了性能。」

#### 動詞位置

**將動詞置於句子早期**，以便更好地解析：

**弱**：「梯度，在被計算和正規化之後，更新了權重。」
**強**：「梯度在被計算和正規化後更新了權重。」

#### 撇號展開（所有格轉換）(Apostrophe Unfolding)

轉換所有格結構以提高清晰度：

**原始**：「X 的 Y (X's Y)」 → **展開**：「X 的 Y (The Y of X)」

**前**：「該模型在測試集上的準確度 (The model's accuracy)」
**後**：「該模型在測試集上的準確度 (The accuracy of the model)」

這並不總是更好，但當句子感覺彆扭時，嘗試展開它。

#### 應刪除的詞彙

在幾乎所有情況下都刪除這些贅詞：
- 「實際上 (actually)」
- 「一點點 (a bit)」
- 「幸運地 (fortunately)」/「不幸地 (unfortunately)」
- 「非常 (very)」/「真的 (really)」
- 「相當 (quite)」
- 「基本上 (basically)」
- 「本質上 (essentially)」
- 過多的連接詞（「然而 (however)」、「此外 (moreover)」、「再者 (furthermore)」，若非必要）

#### 句子構建規則

1. **一個句子一個想法**——如果難以用一個句子表達一個想法，它需要拆成兩個。
2. **無重複音節**——避免在同一個句子中使用發音相似的詞。
3. **每一句都要增加資訊**——刪除只是重複陳述的句子。
4. **始終使用主動語態**——明確行動者（「我們發現……」而不是「被發現……」）。
5. **展開縮寫**——為了正式性，使用「do not」而非「don't」。

#### 段落架構

- **第一句**：清晰地陳述觀點
- **中間句**：用證據支持
- **最後一句**：強化或過渡

不要將關鍵資訊埋在段落中間。

## 用詞選擇與精確度

### 來自 Zachary Lipton 的建議

**消除模稜兩可的詞 (Hedging)**，除非確實存在不確定性：
- 除非必要，否則刪除「可能 (may)」和「可以 (can)」。
- 「提供*非常*緊密的近似」流露出不安全感。
- 「提供緊密的近似」則顯得有信心。

**避免空洞的強化詞**：
- 刪除：非常 (very)、極度 (extremely)、高度 (highly)、顯著 (significantly)（除非是統計學上的）。
- 這些詞傳達的是不安全感，而非力量。

### 來自 Jacob Steinhardt 的建議

**精確優於簡潔**：用具體詞彙替換模糊詞彙。

| 模糊 | 具體 |
|-------|----------|
| 性能 (performance) | 準確度 (accuracy)、延遲 (latency)、吞吐量 (throughput) |
| 提高 (improves) | 準確度提高 X%、延遲降低 Y |
| 大型 (large) | 10 億個參數、1 億個標記 |
| 快速 (fast) | 快 3 倍、50 毫秒延遲 |
| 好結果 (good results) | 92% 準確度、0.85 F1 |

**一致的術語**：用不同的詞稱呼同一個概念會造成混淆。

**選擇一個並堅持使用**：
- 「模型」vs「網路」vs「架構」
- 「訓練」vs「學習」vs「優化」
- 「樣本 (sample)」vs「例子 (example)」vs「實例 (instance)」

### 詞彙信號

**避免暗示增量工作的詞彙**：
- 絕不要用：「結合 (combine)」、「修改 (modify)」、「擴展 (expand)」、「延伸 (extend)」
- 應改用：「開發 (develop)」、「提出 (propose)」、「引入 (introduce)」

**原因**：「我們結合了 X 和 Y」聽起來像是你把兩個現有的想法釘在一起。「我們開發了一種利用 X 來進行 Y 的方法」聽起來像是真正的貢獻。

## 數學寫作

### 來自 Ethan Perez 的建議

為了清晰起見，**展開所有格**：
- 弱：「X 的 Y (X's Y)」
- 強：「X 的 Y (The Y of X)」

範例：「模型的準確度」 → 「模型的準確度 (the accuracy of the model)」

### 一般原則

1. 在定理前**正式陳述所有假設**
2. 在證明旁**提供直覺解釋**
3. 在整篇論文中**使用一致的符號**
4. **在首次使用時定義符號**

### 符號慣例

```latex
% 純量：小寫斜體
$x$, $y$, $\alpha$, $\beta$

% 向量：小寫粗體
$\mathbf{x}$, $\mathbf{v}$

% 矩陣：大寫粗體
$\mathbf{W}$, $\mathbf{X}$

% 集合：大寫花體
$\mathcal{X}$, $\mathcal{D}$

% 函數：羅馬字體用於命名函數
$\mathrm{softmax}$, $\mathrm{ReLU}$
```

## 圖表設計

### 來自 Neel Nanda 的建議

即使讀者跳過文字，圖表也應該講述一個連貫的故事。許多讀者最初確實會跳過文字。

### 設計原則

1. **圖 1 至關重要**：通常是讀者在看過摘要後首先查看的內容。
2. **自給自足的圖說 (Captions)**：讀者應該在沒有主文的情況下也能理解圖表。
3. **圖表內不放標題**：圖說已承擔此功能（ICML/NeurIPS 規則）。
4. **向量圖形**：繪圖使用 PDF/EPS，只有照片才使用 PNG (600 DPI)。

### 無障礙要求

8% 的男性患有色覺障礙。你的圖表必須對他們友善。

**解決方案**：
- 使用色盲友善的調色盤：Okabe-Ito 或 Paul Tol。
- 避免紅綠組合。
- 驗證圖表在灰階下也能運作。
- 除了顏色，還使用不同的線條樣式（實線、虛線、點線）。

## 應避免的常見錯誤

### 結構錯誤

| 錯誤 | 解決方案 |
|---------|----------|
| 緒論太長（>1.5 頁） | 將背景內容移至相關工作 (Related Work) |
| 方法論太晚出現（超過第 3 頁） | 將貢獻前置，削減緒論 |
| 缺少貢獻要點 | 增加 2-4 個具體、可證偽的主張 |
| 實驗沒有明確的主張 | 陳述每個實驗正在測試什麼 |

### 寫作錯誤

| 錯誤 | 解決方案 |
|---------|----------|
| 通用的摘要開場白 | 直接從你的具體貢獻開始 |
| 術語不一致 | 每個概念選擇一個詞彙並堅持使用 |
| 過度使用被動語態 | 使用主動語態：「我們展示」而非「被展示」 |
| 處處都是模糊詞 (Hedging) | 除非確實不確定，否則展現信心 |

### 圖表錯誤

| 錯誤 | 解決方案 |
|---------|----------|
| 繪圖使用點陣圖 | 使用向量圖 (PDF/EPS) |
| 紅綠配色方案 | 使用色盲友善的調色盤 |
| 圖表內放標題 | 將標題放在圖說中 |
| 圖說需要依賴主文 | 讓圖說能自給自足 |

### 引用錯誤

| 錯誤 | 解決方案 |
|---------|----------|
| 逐篇條列式的相關工作 | 按方法論進行組織 |
| 遺漏相關引用 | 審稿人可能就是論文作者——慷慨地引用 |
| AI 生成的引用 | 務必透過 API 驗證 |
| 引用格式不一致 | 使用 BibLaTeX 並保持鍵值 (keys) 一致 |
