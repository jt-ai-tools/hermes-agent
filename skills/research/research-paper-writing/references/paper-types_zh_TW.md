# 非實證類機器學習論文類型

本指南介紹了非標準類型論文的撰寫方法：理論論文（theory papers）、綜述/教學論文（survey/tutorial papers）、基準測試/資料集論文（benchmark/dataset papers）以及立場論文（position papers）。每種類型都有其獨特的結構、證據標準與會議預期。

## 理論論文

### 何時撰寫理論論文

如果您的論文符合以下情況，則應歸類為理論論文：
- 主要貢獻是定理（theorem）、界限（bound）、不可能結果（impossibility result）或正式的特性描述。
- 實驗僅作為輔助驗證，而非核心證據。
- 貢獻在於增進理解，而非追求最先進的（SOTA）效能數據。

### 結構

```
1. 引言 (1-1.5 頁)
   - 問題陳述與動機
   - 主要結果的非正式說明
   - 與先前理論工作的比較
   - 貢獻要點（以非正式方式陳述定理）

2. 預備知識 (0.5-1 頁)
   - 符號表 (Notation table)
   - 正式定義
   - 假設（編號，後續引用）
   - 您所基於的已知結果

3. 主要結果 (2-3 頁)
   - 定理陳述（正式）
   - 證明簡述 (Proof sketches)（直覺說明 + 關鍵步驟）
   - 推論 (Corollaries) 與特殊情況
   - 緊確性 (Tightness) / 最佳性 (Optimality) 的討論

4. 實驗驗證 (1-2 頁，選填但建議)
   - 理論預測是否符合實證行為？
   - 隔離特定現象的合成實驗 (Synthetic experiments)
   - 與先前研究界限（bounds）的比較

5. 相關工作 (1 頁)
   - 理論前輩
   - 您的理論所能解釋的實證研究

6. 討論與開放性問題 (0.5 頁)
   - 您研究結果的侷限性
   - 您的分析所啟發的猜想 (Conjectures)
   - 具體的開放性問題

附錄：
   - 完整證明
   - 技術性引理 (Technical lemmas)
   - 延伸實驗細節
```

### 撰寫定理

**陳述完善的定理模板：**

```latex
\begin{assumption}[Bounded Gradients]\label{assum:bounded-grad}
存在 $G > 0$，使得對於所有 $x \in \mathcal{X}$，都有 $\|\nabla f(x)\| \leq G$。
\end{assumption}

\begin{theorem}[Convergence Rate]\label{thm:convergence}
在假設~\ref{assum:bounded-grad} 和~\ref{assum:smoothness} 下，
使用步長 $\eta = \frac{1}{\sqrt{T}}$ 的演算法~\ref{alg:method} 滿足
\[
\frac{1}{T}\sum_{t=1}^{T} \mathbb{E}\left[\|\nabla f(x_t)\|^2\right]
\leq \frac{2(f(x_1) - f^*)}{\sqrt{T}} + \frac{G^2}{\sqrt{T}}.
\]
特別地，在 $T = O(1/\epsilon^2)$ 次迭代後，我們獲得一個
$\epsilon$-駐點（stationary point）。
\end{theorem}
```

**定理陳述規則：**
- 明確列出所有假設（編號並命名）。
- 包含正式的界限（bound），而不僅是「以 O(·) 速率收斂」。
- 加入白話推論：「特別地，這意味著...」。
- 與已知界限比較：「這比 [先前研究] 的 O(·) 界限改進了...倍」。

### 證明簡述 (Proof Sketches)

對於理論論文而言，證明簡述是正文中最重要的一部分。審稿人會藉此評估您是否具有真正的洞察力，還是僅僅是機械式的推導。

**優秀證明簡述的模式：**

```latex
\begin{proof}[Theorem~\ref{thm:convergence} 的證明簡述]
關鍵洞察在於 [用一句話描述核心概念]。

證明分為三個步驟：
\begin{enumerate}
\item \textbf{分解。} 我們使用 [技術] 將誤差分解為 [項 A]
  與 [項 B]。這將問題簡化為分別界定每一項。

\item \textbf{界定 [項 A]。} 根據 [假設/引理]，[項 A]
  被界定為 $O(\cdot)$。關鍵觀察是 [使得這項工作具有挑戰性的具體洞察]。

\item \textbf{組合。} 選擇 $\eta = 1/\sqrt{T}$ 平衡了兩項，
  從而得出所述的界限。
\end{enumerate}

完整的證明，包括步驟 2 的技術性引理，
請見附錄~\ref{app:proofs}。
\end{proof}
```

**糟糕的證明簡述**：只是用稍微不同的符號重新陳述定理，或者僅說「證明遵循標準技術」。

### 附錄中的完整證明

```latex
\appendix
\section{證明}\label{app:proofs}

\subsection{Theorem~\ref{thm:convergence} 的證明}

我們首先建立兩個技術性引理。

\begin{lemma}[Descent Lemma]\label{lem:descent}
在假設~\ref{assum:smoothness} 下，對於任何步長 $\eta \leq 1/L$：
\[
f(x_{t+1}) \leq f(x_t) - \frac{\eta}{2}\|\nabla f(x_t)\|^2 + \frac{\eta^2 L}{2}\|\nabla f(x_t)\|^2.
\]
\end{lemma}

\begin{proof}
[包含所有步驟的完整證明]
\end{proof}

% 繼續列出剩餘的引理與主定理證明
```

### 理論論文常見陷阱

| 陷阱 | 問題點 | 修正方法 |
|---------|---------|-----|
| 假設過強 | 使結果變得微不足道（trivial） | 討論哪些假設是必要的；證明下界（lower bounds） |
| 未與現有界限比較 | 審稿人無法評估貢獻 | 增加界限對比表 |
| 證明簡述只是縮短版的完整證明 | 未能傳達洞察力 | 專注於 1-2 個關鍵想法；將機械式推導移至附錄 |
| 缺乏實驗驗證 | 審稿人質疑實際相關性 | 增加測試預測結果的合成實驗 |
| 符號不一致 | 令審稿人困惑 | 在「預備知識」章節建立符號表 |
| 捨簡就繁的複雜證明 | 審稿人懷疑有誤 | 優先考慮清晰度而非普遍性 (generality) |

### 理論論文投稿會議

| 會議 | 理論論文接受率 | 備註 |
|-------|----------------------|-------|
| **NeurIPS** | 中等 | 重視具有實際影響力的理論 |
| **ICML** | 高 | 理論組（theory track）實力雄厚 |
| **ICLR** | 中等 | 偏好帶有實證驗證的理論 |
| **COLT** | 高 | 專注於理論的頂級會議 |
| **ALT** | 高 | 演算法學習理論 |
| **STOC/FOCS** | 適用於 TCS 類型的結果 | 如果貢獻主要在組合或演算法方面 |
| **JMLR** | 高 | 無頁數限制；適合長篇證明 |

## 綜述與教學論文

### 何時撰寫綜述論文

- 某個子領域已足夠成熟，需要進行整合。
- 您發現了各個研究之間不為人知的聯繫。
- 該領域的新人缺乏良好的入門指引。
- 自上次綜述以來，該領域的樣貌已發生顯著變化。

**警告**：綜述論文需要真正的專業知識。非該領域專家撰寫的綜述，無論多麼詳盡，都可能遺漏細微差別並誤導讀者。

### 結構

```
1. 引言 (1-2 頁)
   - 定義範圍（包含與排除哪些內容，原因為何）
   - 此時撰寫綜述的動機
   - 組織架構概覽（通常配有一張圖）

2. 背景 / 問題建構 (1-2 頁)
   - 正式問題定義
   - 符號標記（貫穿全篇的一致性）
   - 歷史背景

3. 分類法 (Taxonomy)（核心貢獻）
   - 沿著有意義的軸線組織各種方法
   - 以圖表形式呈現分類法
   - 每個類別都有其對應的小節

4. 詳細內容 (論文主體)
   - 針對每個類別：代表性方法、關鍵概念、優缺點
   - 類別內與跨類別的對比表
   - 不要只是描述——要進行分析與比較

5. 實證比較 (若適用)
   - 標準化的基準測試比較
   - 為所有方法進行公平的超參數調優
   - 雖非總是可行，但能顯著增強綜述的實力

6. 開放性問題與未來方向 (1-2 頁)
   - 該領域應解決的未解之謎
   - 具有潛力但尚未被充分探索的方向
   - 本章節是綜述論文展現真正貢獻之處

7. 結論
```

### 分類法設計

分類法是綜述論文核心的智力貢獻。它應具備：

- **有意義**：類別應對應方法論上的真實差異，而非隨意分組。
- **詳盡無遺**：每篇相關論文都能在其中找到位置。
- **互斥性**（理想情況）：每篇論文主要歸屬於一個類別。
- **具資訊量的命名**：「基於注意力機制的方法」優於「類別 3」。
- **視覺化**：呈現分類法的圖表幾乎總是很有幫助。

**「LLM 推理」綜述的分類軸範例：**
- 按技術分類：思維鏈 (CoT)、思維樹 (ToT)、自我一致性、工具調用。
- 按訓練要求分類：僅提示詞 (Prompting-only)、微調、RLHF。
- 按推理類型分類：數學、常識、邏輯、因果。

### 撰寫標準

- **引用每篇相關論文**——作者會檢查他們的作品是否被包含。
- **公平公正**——不要隨意否定您不偏好的方法。
- **整合而非羅列**——識別模式、權衡取捨與開放性問題。
- **包含對比表**——即使是定性的（功能/特性檢查清單）。
- **投稿前更新**——檢查 arXiv，看自您開始撰寫以來是否有新論文發表。

### 綜述論文投稿管道

| 管道 | 備註 |
|-------|-------|
| **TMLR** (Survey track) | 專門的綜述投稿組；無頁數限制 |
| **JMLR** | 篇幅長，聲望高 |
| **Foundations and Trends in ML** | 邀稿制，但可主動提案 |
| **ACM Computing Surveys** | 廣大的電腦科學讀者群 |
| **arXiv** (獨立發表) | 無同儕審查，但若品質優良則能獲得極高關注度 |
| **會議教學課程 (Tutorials)** | 在 NeurIPS/ICML/ACL 作為教學課程呈現；整理成論文發表 |

## 基準測試與資料集論文

### 何時撰寫基準測試論文

- 現有的基準測試無法衡量您認為重要的面向。
- 出現了新的能力，但尚無標準評估方法。
- 現有基準測試已飽和（所有方法得分皆 >95%）。
- 您希望在碎片化的子領域中標準化評估流程。

### 結構

```
1. 引言
   - 此基準測試填補了什麼評估缺口？
   - 為什麼現有的基準測試不足？

2. 任務定義
   - 正式任務規範
   - 輸入/輸出格式
   - 評估標準（什麼樣的答案才是好答案？）

3. 資料集建構
   - 數據來源與收集方法論
   - 標註流程（若是人工標註）
   - 品質控制措施
   - 資料集統計數據（大小、分佈、分割比例）

4. 基準線 (Baseline) 評估
   - 運行強大的基準線（不要只報告隨機或多數類別結果）
   - 證明該基準測試具有挑戰性但並非不可能
   - 人類表現基準線（若可行）

5. 分析
   - 針對基準線的錯誤分析
   - 是什麼讓項目變得困難或簡單？
   - 建構效度 (Construct validity)：基準測試是否真的衡量了您所聲稱的能力？

6. 預期用途與侷限性
   - 此基準測試應被用於何處？
   - 不應被用於何處？
   - 已知的偏誤或侷限性

7. 資料表 (Datasheet) (附錄)
   - 針對資料集的完整資料表 (Gebru et al.)
```

### 證據標準

審稿人對基準測試論文的評估標準不同於方法論論文：

| 準則 | 審稿人檢查項 |
|-----------|---------------------|
| **評估的新穎性** | 是否衡量了現有基準測試未涵蓋的內容？ |
| **建構效度** | 基準測試是否真的衡量了所述的能力？ |
| **難度校準** | 不會太簡單（飽和）或太困難（表現接近隨機） |
| **標註品質** | 一致性指標、標註員資格、標註指南 |
| **文件記錄** | 資料表、授權協議、維護計劃 |
| **重現性** | 他人是否能輕易使用此基準測試？ |
| **倫理考量** | 偏誤分析、同意書、敏感內容處理 |

### 資料集文件記錄（必要）

遵循「資料集資料表」(Datasheets for Datasets) 框架 (Gebru et al., 2021)：

```
資料表問題：
1. 動機
   - 為何建立此資料集？
   - 由誰代表誰建立？
   - 誰資助了這項工作？

2. 組成
   - 實例代表什麼？
   - 共有多少實例？
   - 包含所有可能的實例還是一個樣本？
   - 是否有標籤？若有，如何確定的？
   - 是否有建議的數據分割比例？

3. 收集過程
   - 如何收集數據？
   - 誰參與了收集？
   - 時間範圍為何？
   - 是否進行過倫理審查？

4. 預處理
   - 進行了哪些預處理？
   - 是否儲存了「原始」數據？

5. 用途
   - 已被用於哪些任務？
   - 不應被用於何處？
   - 還能用於其他哪些任務？

6. 發佈
   - 如何發佈？
   - 使用何種授權協議？
   - 是否有任何限制？

7. 維護
   - 誰負責維護？
   - 使用者如何聯繫維護者？
   - 是否會更新？如何更新？
   - 是否有勘誤表 (erratum)？
```

## 立場論文 (Position Papers)

### 何時撰寫立場論文

- 您對該領域的發展方向有獨到的見解 or 論點。
- 您想要挑戰一個被廣泛接受的假設。
- 您想要基於分析提出一個研究議程 (research agenda)。
- 您發現了當前方法論中存在的系統性問題。

### 結構

```
1. 引言
   - 在第一段清晰地陳述您的論點 (thesis)
   - 為什麼現在這個議題很重要

2. 背景
   - 領域現狀
   - 您所挑戰的主流假設

3. 論證
   - 提出您的論點並輔以證據
   - 證據可以是：實證數據、理論分析、邏輯論證、
     個案研究、歷史先例
   - 保持嚴謹——這不是一篇隨筆

4. 反方論點
   - 認真對待最強有力的反駁
   - 解釋為什麼它們不會動搖您的論點
   - 在適當的地方做出讓步——這能增強公信力

5. 影響
   - 該領域應做出哪些改變？
   - 您的論點所啟發的具體研究方向
   - 評估/方法論應如何改變？

6. 結論
   - 重申論點
   - 呼籲採取行動
```

### 撰寫標準

- **在開頭就拋出最強有力的論點**——不要在第一段就含糊其詞。
- **誠實地面對反方論點**——優秀的立場論文應處理最強大的反對意見，而非專挑弱點打。
- **提供證據**——沒有證據的立場論文僅僅是社論。
- **具體明確**——「領域應該執行 X」優於「需要更多研究」。
- **不要斷章取義 (Straw-man)**——公平地描述對方的立場。

### 立場論文投稿管道

| 管道 | 備註 |
|-------|-------|
| **ICML** (Position track) | 專門的立場論文組別 |
| **NeurIPS** (Workshop papers) | 工作坊通常歡迎立場類文章 |
| **ACL** (Theme papers) | 當您的立場與會議主題一致時 |
| **TMLR** | 接受論證嚴謹的立場論文 |
| **CACM** | 針對廣大的電腦科學讀者群 |

## 可重現性與複製研究論文 (Reproducibility and Replication Papers)

### 何時撰寫可重現性論文

- 您嘗試複製已發表的結果並成功或失敗。
- 您想要在不同條件下驗證研究聲明。
- 您發現某個流行方法的效能取決於未被報告的細節。

### 結構

```
1. 引言
   - 您正在複製哪篇論文/結果？
   - 為什麼這次複製研究具有價值？

2. 原始聲明
   - 陳述原始論文中的確切聲明
   - 提供了哪些證據？

3. 方法論
   - 您的複製方法
   - 與原始版本的差異（若有）及其原因
   - 原始論文中遺漏了哪些資訊？

4. 結果
   - 與原始結果的並排對比
   - 統計比較（信賴區間是否重疊？）
   - 哪些部分被成功複製，哪些沒有？

5. 分析
   - 若結果不同：原因為何？哪些部分是敏感的？
   - 隱藏的超參數或實現細節？
   - 對種子、硬體、函式庫版本的魯棒性為何？

6. 建議
   - 給原作者：應澄清哪些內容？
   - 給從業者：應注意什麼？
   - 給該領域：從中獲得了哪些可重現性的教訓？
```
