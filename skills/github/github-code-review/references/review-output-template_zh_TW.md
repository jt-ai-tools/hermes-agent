# 審閱輸出範本 (Review Output Template)

將此作為 PR 審閱摘要註解的結構。複製並填寫各個部分。

## 用於 PR 摘要註解 (PR Summary Comment)

```markdown
## 程式碼審閱摘要 (Code Review Summary)

**裁決：[已批准 ✅ | 要求變更 🔴 | 已審閱 💬]** ([N] 個問題, [N] 個建議)

**PR:** #[編號] — [標題]
**作者:** @[使用者名稱]
**變更檔案:** [N] (+[新增] -[刪除])

### 🔴 嚴重 (Critical)
<!-- 在合併前「必須」修復的問題 -->
- **檔案.py:行號** — [描述]。建議：[修復方案]。

### ⚠️ 警告 (Warnings)
<!-- 「應該」修復，但非嚴格強制阻擋的問題 -->
- **檔案.py:行號** — [描述]。

### 💡 建議 (Suggestions)
<!-- 非強制性的改進、風格偏好、未來考量 -->
- **檔案.py:行號** — [描述]。

### ✅ 看起來不錯 (Looks Good)
<!-- 指出做得好的部分 —— 正向回饋 -->
- [做得好的方面]

---
*由 Hermes Agent 審閱*
```

## 嚴重程度指南 (Severity Guide)

| 級別 | 圖示 | 何時使用 | 是否阻擋合併？ |
|-------|------|-------------|---------------|
| 嚴重 (Critical) | 🔴 | 安全漏洞、資料遺失風險、當機、核心功能受損 | 是 |
| 警告 (Warning) | ⚠️ | 非核心路徑的錯誤、缺少錯誤處理、新程式碼缺少測試 | 通常是 |
| 建議 (Suggestion) | 💡 | 風格改進、重構想法、效能提示、文件缺漏 | 否 |
| 看起來不錯 (Looks Good) | ✅ | 乾淨的模式、良好的測試覆蓋率、清晰的命名、聰明的設計決策 | 不適用 |

## 裁決決定 (Verdict Decision)

- **已批准 ✅ (Approved)** — 零嚴重/警告項目。僅有建議或完全沒問題。
- **要求變更 🔴 (Changes Requested)** — 存在任何嚴重或警告項目。
- **已審閱 💬 (Reviewed)** — 僅為觀察（草稿 PR、不確定的發現、資訊性內容）。

## 用於行內註解 (Inline Comments)

在行內註解前加上嚴重程度圖示，以便於快速瀏覽：

```
🔴 **嚴重：** 使用者輸入直接傳遞給 SQL 查詢 —— 請使用參數化查詢以防止注入。
```

```
⚠️ **警告：** 此錯誤被靜默吞沒。至少應將其記錄在日誌 (log) 中。
```

```
💡 **建議：** 這可以使用字典推導式 (dict comprehension) 來簡化：
`{k: v for k, v in items if v is not None}`
```

```
✅ **讚：** 此處上下文管理器 (context manager) 使用得很好 —— 確保在發生異常時進行清理。
```

## 用於本地 (推送前) 審閱

在推送前進行本地審閱時，使用相同的結構，但將其作為發送給使用者的訊息而非 PR 註解。跳過 PR 元數據標頭，直接從嚴重程度部分開始。
