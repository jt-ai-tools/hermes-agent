# 持久化記憶

Hermes Agent 具有跨工作階段持久化的有界、精選記憶。這讓它能記住您的偏好、您的專案、您的環境以及它所學到的內容。

## 運作方式

代理的記憶由兩個檔案組成：

| 檔案 | 用途 | 字元限制 |
|------|---------|------------|
| **MEMORY.md** | 代理的個人筆記 — 環境事實、慣例、學到的內容 | 2,200 字元 (約 800 個權杖) |
| **USER.md** | 使用者個人資料 — 您的偏好、溝通風格、期望 | 1,375 字元 (約 500 個權杖) |

這兩個檔案都儲存在 `~/.hermes/memories/` 中，並在工作階段開始時以凍結快照的形式注入到系統提示中。代理透過 `memory` 工具管理自己的記憶 — 它可以新增、替換或移除項目。

:::info
字元限制使記憶保持集中。當記憶已滿時，代理會整合或替換項目，以騰出空間存放新資訊。
:::

## 記憶如何出現在系統提示中

在每個工作階段開始時，記憶項目會從磁碟載入，並作為凍結區塊呈現到系統提示中：

```
══════════════════════════════════════════════
MEMORY (您的個人筆記) [67% — 1,474/2,200 字元]
══════════════════════════════════════════════
使用者的專案是一個位於 ~/code/myapi 的 Rust 網路服務，使用 Axum + SQLx
§
這台機器執行 Ubuntu 22.04，安裝了 Docker 和 Podman
§
使用者偏好簡潔的回應，不喜歡冗長的解釋
```

格式包括：
- 顯示哪個儲存庫（MEMORY 或 USER PROFILE）的標頭
- 使用百分比和字元數，讓代理了解容量
- 由 `§`（章節符號）分隔符隔開的個別項目
- 項目可以是多行的

**凍結快照模式 (Frozen snapshot pattern)：** 系統提示注入在工作階段開始時捕捉一次，且在工作階段期間絕不更改。這是刻意設計的 — 它可以保留 LLM 的前綴快取 (prefix cache) 以提高效能。當代理在工作階段期間新增/移除記憶項目時，更改會立即持久化到磁碟，但在下一個工作階段啟動前不會出現在系統提示中。工具回應始終顯示即時狀態。

## 記憶工具操作

代理使用 `memory` 工具執行以下操作：

- **add** — 新增一條記憶項目
- **replace** — 使用更新後的內容替換現有項目（透過 `old_text` 使用子字串匹配）
- **remove** — 移除不再相關的項目（透過 `old_text` 使用子字串匹配）

沒有 `read` 操作 — 記憶內容在工作階段開始時會自動注入系統提示。代理將其記憶視為其對話上下文的一部分。

### 子字串匹配 (Substring Matching)

`replace` 和 `remove` 操作使用短且唯一的子字串匹配 — 您不需要完整的項目文字。`old_text` 參數只需要是一個能精確識別一個項目的唯一子字串：

```python
# 如果記憶包含 "User prefers dark mode in all editors"
memory(action="replace", target="memory",
       old_text="dark mode",
       content="User prefers light mode in VS Code, dark mode in terminal")
```

如果子字串匹配到多個項目，則會返回錯誤，要求提供更精確的匹配。

## 兩個目標說明

### `memory` — 代理的個人筆記

用於代理需要記住的關於環境、工作流程和學到的教訓等資訊：

- 環境事實（作業系統、工具、專案結構）
- 專案慣例和配置
- 發現的工具特性和解決方法
- 已完成任務的日記條目
- 行之有效的技能和技術

### `user` — 使用者個人資料

用於關於使用者的身份、偏好和溝通風格等資訊：

- 姓名、角色、時區
- 溝通偏好（簡潔 vs 詳細，格式偏好）
- 忌諱和要避免的事項
- 工作流程習慣
- 技術能力水準

## 要儲存 vs 略過的內容

### 儲存這些內容（主動儲存）

代理會自動儲存 — 您不需要詢問。它會在學到以下內容時儲存：

- **使用者偏好：** 「我偏好 TypeScript 而非 JavaScript」→ 儲存到 `user`
- **環境事實：** 「這台伺服器執行 Debian 12 並安裝 PostgreSQL 16」→ 儲存到 `memory`
- **修正：** 「不要對 Docker 指令使用 `sudo`，使用者已在 docker 群組中」→ 儲存到 `memory`
- **慣例：** 「專案使用 tabs，120 字元行寬，Google 風格的文件字串」→ 儲存到 `memory`
- **完成的工作：** 「於 2026-01-15 將資料庫從 MySQL 遷移到 PostgreSQL」→ 儲存到 `memory`
- **明確要求：** 「記住我的 API 金鑰輪換是每月進行一次」→ 儲存到 `memory`

### 略過這些內容

- **瑣碎/明顯的資訊：** 「使用者詢問了關於 Python 的問題」— 太過模糊而無用
- **容易重新發現的事實：** 「Python 3.12 支援 f-string 巢狀」— 可以透過網路搜尋找到
- **原始數據轉儲：** 大型程式碼區塊、日誌檔案、數據表 — 對記憶來說太大了
- **工作階段特定的臨時資訊：** 臨時檔案路徑、一次性的偵錯上下文
- **上下文檔案中已有的資訊：** SOUL.md 和 AGENTS.md 的內容

## 容量管理

記憶有嚴格的字元限制，以保持系統提示在有界範圍內：

| 儲存庫 | 限制 | 典型項目數 |
|-------|-------|----------------|
| memory | 2,200 字元 | 8-15 個項目 |
| user | 1,375 字元 | 5-10 個項目 |

### 當記憶已滿時會發生什麼事

當您嘗試新增一個會超過限制的項目時，工具會返回錯誤：

```json
{
  "success": false,
  "error": "Memory at 2,100/2,200 chars. Adding this entry (250 chars) would exceed the limit. Replace or remove existing entries first.",
  "current_entries": ["..."],
  "usage": "2,100/2,200"
}
```

代理接著應該：
1. 讀取目前的項目（顯示在錯誤回應中）
2. 識別可以移除或整合的項目
3. 使用 `replace` 將相關項目合併為較短的版本
4. 然後 `add` 新項目

**最佳實踐：** 當記憶容量超過 80% 時（可在系統提示標頭中看到），在新增新項目之前先整合項目。例如，將三個獨立的「專案使用 X」項目合併為一個全面的專案描述項目。

### 良好記憶項目的實際範例

**緊湊、資訊密集型項目效果最好：**

```
# 好：封裝了多個相關事實
使用者執行 macOS 14 Sonoma，使用 Homebrew，安裝了 Docker Desktop 和 Podman。Shell：zsh 與 oh-my-zsh。編輯器：VS Code 與 Vim 鍵位綁定。

# 好：具體、可執行的慣例
專案 ~/code/api 使用 Go 1.22、sqlc 進行資料庫查詢、chi 路由。使用 'make test' 執行測試。透過 GitHub Actions 進行 CI。

# 好：帶有上下文的經驗教訓
預發布伺服器 (10.0.1.50) 需要 SSH 連接埠 2222，而非 22。金鑰位於 ~/.ssh/staging_ed25519。

# 壞：太模糊
使用者有一個專案。

# 壞：太冗長
在 2026 年 1 月 5 日，使用者要求我查看他們的專案，該專案
位於 ~/code/api。我發現它使用 Go 版本 1.22，而且...
```

## 重複防止

記憶系統會自動拒絕完全重複的項目。如果您嘗試新增已存在的內容，它會返回成功並顯示「未新增重複內容」的訊息。

## 安全掃描

記憶項目在被接受之前會進行注入和外洩模式掃描，因為它們會被注入到系統提示中。符合威脅模式（提示詞注入、憑證外洩、SSH 後門）或包含不可見 Unicode 字元的內容將被封鎖。

## 工作階段搜尋

除了 MEMORY.md 和 USER.md 之外，代理還可以使用 `session_search` 工具搜尋過去的對話：

- 所有 CLI 和通訊工作階段都儲存在 SQLite (`~/.hermes/state.db`) 中，並具備 FTS5 全文搜尋功能
- 搜尋查詢會返回相關的過去對話，並附帶 Gemini Flash 的摘要
- 代理可以找到幾週前討論過的內容，即使這些內容不在其活動記憶中

```bash
hermes sessions list    # 瀏覽過去的工作階段
```

### session_search vs memory

| 功能 | 持久化記憶 | 工作階段搜尋 |
|---------|------------------|----------------|
| **容量** | 總計約 1,300 個權杖 | 無限制（所有工作階段） |
| **速度** | 即時（在系統提示中） | 需要搜尋 + LLM 摘要 |
| **使用案例** | 關鍵事實始終可用 | 尋找特定的過去對話 |
| **管理** | 由代理手動精選 | 自動 — 儲存所有工作階段 |
| **權杖成本** | 每個工作階段固定（約 1,300 個權杖） | 按需（需要時搜尋） |

**記憶 (Memory)** 用於始終應存在於上下文中的關鍵事實。**工作階段搜尋 (Session search)** 用於「我們上週是否討論過 X？」之類的查詢，代理需要回憶過去對話中的細節。

## 組態設定

```yaml
# 位於 ~/.hermes/config.yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200   # 約 800 個權杖
  user_char_limit: 1375     # 約 500 個權杖
```

## 外部記憶提供者

對於超出 MEMORY.md 和 USER.md 的更深層、持久化記憶，Hermes 附帶了 8 個外部記憶提供者外掛程式 — 包括 Honcho、OpenViking、Mem0、Hindsight、Holographic、RetainDB、ByteRover 和 Supermemory。

外部提供者與內建記憶**並行**執行（絕不替換它），並增加了知識圖譜、語義搜尋、自動事實提取和跨工作階段使用者建模等功能。

```bash
hermes memory setup      # 選擇一個提供者並進行配置
hermes memory status     # 檢查哪些提供者處於活動狀態
```

有關每個提供者的完整詳細資訊、設置說明和比較，請參閱 [記憶提供者](./memory-providers.md) 指南。
