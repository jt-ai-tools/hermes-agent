# 1-3-1 溝通原則

一種結構化決策格式，適用於任務有多種可行方法且使用者需要明確建議的情形。它會產生簡潔的問題框架、三個帶有權衡分析的選項，以及針對建議路徑的可執行計畫。

## 何時使用

- 使用者明確要求 "1-3-1" 回應。
- 使用者針對技術決策說「給我選項」或「我有什麼選擇」。
- 任務有多種可行方法，且具有實質性的權衡（架構、工具、遷移策略）。
- 使用者需要一份可以轉發給團隊或利害關係人的提案。

對於只有一個明顯答案的簡單問題、除錯過程或使用者已經決定方法的任務，請**不要**使用此技能。

## 程序

1. **問題 (Problem)** (一句話)
   - 用一句簡潔的話陳述核心決策或期望結果。
   - 專注於「什麼」(what)，而非「如何」(how) — 不含實作細節、工具名稱或特定技術。
   - 保持精簡。如果你需要用到「而且」，那麼你可能正在描述兩個問題。

2. **選項 (Options)** (正好三個)
   - 提供三個獨特且可行的策略，標記為 A、B、C。
   - 每個選項都包含簡短描述、優點和缺點。
   - 選項應代表真正不同的策略，而非同一方法的細微變化。

3. **建議 (Recommendation)** (一個選項)
   - 陳述你建議哪個選項以及原因，並根據使用者的背景和優先順序進行分析。
   - 直接了當 — 這是你的專業判斷，而非模糊的言論。

4. **完成定義 (Definition of Done)**
   - 列出建議選項的具體成功標準。
   - 這些應是具體、可驗證的結果 — 而非模糊的願望。
   - 如果使用者選擇了不同的選項，請修改此部分以進行匹配。

5. **實作計畫 (Implementation Plan)**
   - 執行建議選項的具體步驟。
   - 在適用時包含特定的命令、工具或行動。
   - 如果使用者選擇了不同的選項，請修改此部分以進行匹配。

## 驗證

- 回應中正好有一個「問題」句子。
- 回應中正好有三個選項 (A, B, C)，且每個選項都有優缺點。
- 回應中有一個挑選其中一個選項並給出理由的「建議」。
- 「完成定義」和「實作計畫」與建議選項保持一致。
- 如果使用者選擇了不同的選項，「建議」、「完成定義」和「實作計畫」會相應更新。

## 範例

使用者：「為我們的 API 客戶端加入重試邏輯提供一個 1-3-1 提案。」

代理人：

**問題：** 外部服務的 API 呼叫在負載下會間歇性失敗，需要優雅的重試處理機制。

**選項：**

- **選項 A：在共享公用程式中使用帶有抖動 (jitter) 的指數退避 (exponential backoff)。**
  優點：簡單且易於理解的模式；單一實作可供所有呼叫者共享。
  缺點：沒有熔斷機制 (circuit-breaking)；呼叫者在重試期間仍會阻塞。

- **選項 B：在每個端點呼叫上使用重試函式庫 (tenacity) 作為裝飾器。**
  優點：可針對每個端點進行設定；內建支援熔斷器和速率限制。
  缺點：增加了依賴項；裝飾器模式可能會模糊控制流程。

- **選項 C：具備死信 (dead letter) 處理功能的基於佇列的非同步重試。**
  優點：非阻塞；失敗的呼叫會被追蹤且可重放；擴展性好。
  缺點：複雜度顯著增加；需要佇列後端；對於低流量服務而言過於大材小用。

**建議：** 選項 A。呼叫量尚不足以證明需要佇列基礎設施，而具備指數退避 + 抖動的共享公用程式可以乾淨地涵蓋失敗模式，且無需增加新的依賴項。

**完成定義：**
- `api_retry()` 公用程式可處理暫時性的 HTTP 錯誤 (429, 502, 503, 504)。
- 最多重試 3 次，並採用指數退避與抖動。
- 所有外部 API 呼叫都透過該重試公用程式進行。
- 記錄每次重試嘗試，包含端點和嘗試次數。
- 測試涵蓋重試耗盡、立即成功以及先暫時失敗後成功的情境。

**實作計畫：**
1. 建立 `utils/api_retry.py`，並具備可設定的最大重試次數、基礎延遲和可重試的狀態碼。
2. 使用 `random.uniform(0, base_delay)` 加入抖動，以防止群聚效應 (thundering herd)。
3. 在 `api_client.py` 中使用重試公用程式包裝現有的 API 呼叫。
4. 加入單元測試，為每個重試情境模擬 HTTP 回應。
5. 在負載下針對不穩定的模擬端點進行簡單的壓力測試以進行驗證。
