# 調查範本

針對常見供應鏈攻擊場景預先建構的假設與調查範本。
每個範本包含：攻擊模式、關鍵證據收集清單以及假設啟動器。

---

## 範本 1：維護者帳號遭入侵

**模式**：攻擊者獲得合法維護者帳號的存取權限（透過釣魚、憑證填充等方式），並利用該帳號推送惡意程式碼、建立帶有後門的發行版本或竊取 CI 密鑰。

**真實案例**：XZ Utils (2024), Codecov (2021), event-stream (2018)

**關鍵證據收集清單**：
- [ ] 來自維護者帳號在正常工作時間/時區之外的推送事件（Push events）
- [ ] 新增依賴項、混淆程式碼或修改建置腳本的提交（Commits）
- [ ] 在可疑推送後立即建立發行版本（以最大化軟體包分發）
- [ ] 新增未知協作者的成員事件（MemberEvent）（攻擊者新增備用存取權限）
- [ ] 具有意外密鑰存取或類似資料外洩行為的工作流執行事件（WorkflowRunEvent）
- [ ] 帳號登入位置變更（檢查社群媒體、會議演講以進行證實）

**假設啟動器**：
```
[假設] 根據異常的提交時間 [EV-XXXX] 和地理存取模式 [EV-YYYY]，執行者 <HANDLE> 的帳號在 <DATE> 前後遭到入侵。
```
```
[假設] 發行版本 <VERSION> 是由受侵害帳號發布的，目的是向下游用戶推送惡意程式碼，證據是在發布前 <N> 小時新增了惡意提交 [EV-XXXX] [EV-YYYY]。
```

## 範本 2：惡意依賴項注入

**模式**：修改信任的軟體包以在依賴項中包含惡意程式碼，或將新的惡意依賴項注入現有軟體包。

**關鍵證據收集清單**：
- [ ] 可疑提交前後 `package.json`/`requirements.txt`/`go.mod` 的差異（Diff）
- [ ] 新依賴項的發布時間戳記與注入提交的時間戳記對比
- [ ] 新依賴項是否存在於 npm/PyPI 以及其所有者是誰
- [ ] 注入的依賴項程式碼中是否有任何混淆模式
- [ ] 在安裝時執行程式碼的安裝時腳本（`postinstall`, `setup.py` 等）

**假設啟動器**：
```
[假設] 提交 <SHA> [EV-XXXX] 引入了依賴項 <PACKAGE@VERSION>，這似乎是由執行者 <HANDLE> 發布的惡意軟體包 [EV-YYYY]，旨在於安裝期間執行 <行為>。
```

## 範本 3：CI/CD 流水線注入

**模式**：攻擊者修改 GitHub Actions 工作流以竊取密鑰、外洩程式碼或在建置輸出中注入惡意構件。

**關鍵證據收集清單**：
- [ ] 可疑期間前後所有 `.github/workflows/*.yml` 文件的差異（Diff）
- [ ] 由修改後的工作流觸發的工作流執行事件（WorkflowRunEvents）
- [ ] 工作流步驟中新增的任何 `curl`、`wget` 或網路調用
- [ ] 引用 `secrets.*` 的新或修改後的 `env:` 區塊
- [ ] 修改後的工作流執行所產生的構件（Artifacts）

**假設啟動器**：
```
[假設] 工作流文件 <FILE> 在提交 <SHA> [EV-XXXX] 中被修改，以透過 <方法> 竊取儲存庫密鑰，這可以從新增的網路調用模式 [EV-YYYY] 中得到證實。
```

## 範本 4：拼寫錯誤劫持（Typosquatting）/ 依賴項混淆

**模式**：攻擊者註冊一個名稱與熱門軟體包（或內部軟體包名稱）相似的軟體包，以攔截拼錯名稱的用戶的安裝。

**關鍵證據收集清單**：
- [ ] 該可疑軟體包在註冊表上的註冊時間戳記
- [ ] 軟體包內容：包含惡意程式碼還是僅為存根（stub）？
- [ ] 該可疑軟體包的下載統計數據
- [ ] 可能成為目標的內部軟體包名稱（如果是私有儲存庫範疇）
- [ ] 惡意軟體包元數據中對合法軟體包的任何引用

**假設啟動器**：
```
[假設] 軟體包 <MALICIOUS_NAME> 於 <DATE> [EV-XXXX] 註冊，目的是對 <LEGITIMATE_NAME> 進行拼寫錯誤劫持，針對拼錯軟體包名稱的用戶。該軟體包包含 <行為> [EV-YYYY]。
```

## 範本 5：強制推送歷史重寫（證據抹除）

**模式**：在偵測到惡意提交後（或在引起廣泛關注前），攻擊者進行強制推送（force-push）以從分支歷史中移除惡意提交。

**偵測是關鍵** —— 此範本著重於證明抹除行為確實發生過。

**關鍵證據收集清單**：
- [ ] GH Archive 中 `distinct_size=0` 的 PushEvent（強制推送指標） [EV-XXXX]
- [ ] 強制推送前提交的 SHA（來自 GH Archive `payload.before`）
- [ ] 透過直接 URL 或 `git fetch origin SHA` 恢復已抹除的提交
- [ ] 抹除前提交頁面的 Wayback Machine 快照
- [ ] git log 中的時間軸缺口（Archive 中可見 N 個提交，但當前儲存庫中僅 M < N 個）

**假設啟動器**：
```
[假設] 執行者 <HANDLE> 於 <DATE> [EV-XXXX] 強制推送了分支 <BRANCH>，以抹除包含 <惡意內容> 的提交 <SHA> [EV-YYYY]。已抹除的提交已透過 <方法> 恢復 [EV-ZZZZ]。
```

## 交叉調查檢查清單

無論使用哪種範本，請應用於每項調查：

- [ ] 檢查所有貢獻者中是否有新建立的帳號（在惡意活動發生時建立未滿 30 天）
- [ ] 檢查該期間是否有任何維護者帳號更改了電子郵件（帳號接管的跡象）
- [ ] 驗證可疑提交上的 GPG 簽署是否與已知的維護者密鑰匹配
- [ ] 檢查儲存庫是否在事件發生前後變更了所有權或轉移了組織
- [ ] 尋找緊接在惡意提交之後的「清理」提交（掩蓋模式）
- [ ] 檢查同一作者的相關軟體包/儲存庫是否有類似模式
